The handshake (TLS 1.3 — one round trip)

1 · ClientHello
2 · ServerHello
3 · Certificate
4 · CertVerify
5 · Finished (S)
6 · Finished (C)

1–2 negotiate + swap key shares · 3–4 prove identity · 5–6 confirm integrity. Messages 2–5 arrive in one server flight.

1 ClientHellooffered versions/ciphers + client key share
2 ServerHellochosen version/cipher + server key share
3 Certificateserver's certificate chain (encrypted)
4 CertificateVerifysignature proving the server holds the private key
5 Finished (server)MAC over the handshake
6 Finished (client)client MAC — channel open, app data flows

Key idea: the session key is never sent — both sides derive it from the two key shares (ephemeral Diffie–Hellman).

The four certificate checks

Chain → trusted root
Hostname matches
Not expired
Not revoked

Failure → cause → fix

curl says… Cause Fix
certificate has expired Validity window passed (not-after). Renew / reissue — check the auto-renewal job.
self-signed certificate Leaf signs itself; no CA above it. Get a CA-issued cert (or trust it in dev only).
…subject name matches… Hostname not in the cert's SAN list. Reissue covering the name, or use a listed name.
unable to get local issuer Missing intermediate, or untrusted/internal root. Server: send full chain. Client: add internal root.

Works in the browser but fails in curl? Almost always a missing intermediate the browser cached and curl didn't.

Commands

inspect a live connection 
# Narrate the handshake, message by message
curl -v https://example.com

# Full cert chain + validity dates + names
openssl s_client -connect example.com:443 -servername example.com

# Just the validity window of the presented cert
openssl s_client -connect example.com:443 -servername example.com \
  </dev/null 2>/dev/null | openssl x509 -noout -dates

# Just the names the cert is valid for (SAN)
openssl s_client -connect example.com:443 -servername example.com \
  </dev/null 2>/dev/null | openssl x509 -noout -ext subjectAltName
