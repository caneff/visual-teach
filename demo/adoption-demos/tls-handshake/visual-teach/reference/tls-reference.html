1Glossary

The canonical vocabulary for this course. Every lesson uses these terms as defined here.

TLS
Transport Layer Security. The protocol that turns a plain TCP connection into a private, tamper-evident, authenticated one — the S in HTTPS.
Confidentiality
No eavesdropper on the path can read the bytes.
Integrity
No one can alter the bytes in flight without both sides detecting it.
Authentication
The client gets proof it is really talking to the named server, not an impostor. (In normal web TLS only the server is authenticated.)
Client
The side that opens the connection — your browser or curl.
Server
The side being connected to; it presents a certificate.
Certificate
A public key bound to an identity (a hostname), signed by a CA. It says "this public key belongs to example.com."
Public / private key
A keypair where one key undoes the other. Treated here as a black box: anyone can hold the public key; only the owner holds the private key.
CA (Certificate Authority)
A trusted third party that signs certificates after checking the requester controls the identity. Roots ship pre-trusted in your OS and browser.
Trust store
The set of root CA certificates your OS/browser already trusts. The anchor every chain must reach.
Chain
Leaf (the server's cert) → one or more intermediates → a root in the trust store. Each link is signed by the next one up.
ClientHello
First handshake message. Offers TLS versions and cipher suites, a random nonce, the target hostname (SNI), and a Diffie–Hellman key share.
ServerHello
Reply choosing the version and cipher, with the server's own key share. In TLS 1.3 everything after it is already encrypted.
Key share
A public Diffie–Hellman value. The two shares let each side compute the same secret without it ever crossing the wire.
Ephemeral Diffie–Hellman
A fresh, throw-away key exchange per connection. Gives forward secrecy: a later key compromise can't decrypt past traffic.
Session key
The shared symmetric key both sides derive from the exchange and use to encrypt the actual data.
Finished
The last handshake message from each side: a MAC over the whole transcript proving nothing was tampered with and both derived the same keys.
SNI (Server Name Indication)
The hostname the client puts in the ClientHello so a server hosting many sites knows which certificate to send.
Self-signed
A certificate signed by its own key instead of a CA — no external party vouches for it, so clients don't trust it by default.
Revocation
A CA declaring a still-unexpired certificate invalid (e.g. its key leaked), checked via CRL or OCSP.

2Handshake cheat-sheet

The TLS 1.3 full handshake — one round trip. Read top to bottom; by the ServerHello onward everything is encrypted.

Application data

One round trip, then encrypted application data flows.

Message Sent by Carries Proves / does
ClientHello Client Versions, cipher suites, random, SNI, key share Opens the negotiation; offers a half of the key exchange
ServerHello Server Chosen version & cipher, random, key share Fixes the parameters; completes the key exchange inputs
Certificate Server The server's cert chain (leaf → intermediates) Lets the client check identity against a trusted root
CertificateVerify Server A signature over the transcript Proves the server holds the cert's private key
Finished Both A MAC over the whole transcript Proves integrity and that both derived the same keys
The one idea — Each side sends a public key share. From my private + your public, both compute the same secret; an eavesdropper sees only the two public shares and cannot. The session key is never transmitted.

4Failure diagnosis

Map a real error to which of the four checks failed, and what to do. The five live test hosts are subdomains of badssl.com.

Symptom / error Check that failed What it means What to do
Certificate has expired Validity window Cert is past notAfter — usually a missed renewal Renew/reissue the cert; check the server clock
Self-signed certificate Chains to root No CA vouches for it; signed by its own key Use a CA-issued cert, or trust it deliberately for dev only
Unable to get local issuer / unknown CA Chains to root Chain doesn't reach a trusted root (missing intermediate, or private CA) Install the intermediate on the server; add the root to the trust store if it's yours
Hostname / wrong host mismatch Hostname The URL host isn't in the cert's names Use the name the cert covers, or reissue with the right SAN
Certificate revoked Revocation The CA invalidated it (key compromise, mis-issuance) Get a fresh cert; never bypass a revocation
Inspect, don't guess — Read the live chain and dates with openssl s_client -connect host:443 -servername host or your browser's certificate viewer before changing anything.
