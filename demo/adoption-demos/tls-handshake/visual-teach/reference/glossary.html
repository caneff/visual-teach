The spine of the course: a client runs a handshake, receives a certificate, and walks its chain of trust to a root it already holds.

Guarantees & actors

TLSTransport Layer Security — the protocol that secures HTTPS. Current version: 1.3 (RFC 8446).
ConfidentialityGuarantee that no one in the middle can read the bytes.
IntegrityGuarantee that no one can change the bytes undetected.
AuthenticationGuarantee you're talking to the server you named, not an impostor.
ClientThe party that opens the connection and runs the certificate checks (your browser, curl).
ServerThe party proving its identity; holds a private key and a certificate.
Certificate Authority (CA)Trusted third party that signs certificates. Not online during your connection — it signed earlier.

Keys & certificates

KeypairA public key + its matching private key. One key undoes the other.
Public keyShareable half of a keypair; can be published freely.
Private keySecret half; never leaves its owner. Proving you hold it proves identity.
SignatureA value only the private-key holder could produce, verifiable by anyone with the public key.
CertificateA public key bound to an identity (hostname), signed by a CA the client trusts.
LeafThe end-entity certificate the server presents for its hostname.
IntermediateA CA certificate between the leaf and the root; signs leaves so the root can stay offline.
RootThe top of the chain; its certificate lives in the client's trust store.
Chain of trustLeaf → intermediate → root. Valid only if it ends at a trusted root.
Trust storeThe set of root CAs a client ships with and trusts in advance.
SANSubject Alternative Name — the list of hostnames a certificate is valid for.
Validity windowThe not-before / not-after dates outside which a cert is rejected.
RevocationA CA pulling a cert before it expires; checked via OCSP or a CRL.
Self-signedA leaf that signs itself — no CA above it, so the chain check fails.

The handshake

HandshakeThe message exchange that sets up the secure channel — one round trip in TLS 1.3.
Cipher suiteThe agreed bundle of algorithms (e.g. TLS_AES_256_GCM_SHA384) for encryption and integrity.
ClientHelloMessage 1. Offered versions & ciphers + the client's key share.
ServerHelloMessage 2. Chosen version & cipher + the server's key share.
Key shareA public Diffie–Hellman value sent in a Hello; combined with a secret to derive the session key.
Ephemeral Diffie–HellmanKey agreement where both sides derive the same secret from exchanged public values, using throwaway secrets.
Session keyThe shared symmetric key both sides compute independently; never transmitted.
Certificate (message)Message 3. The server's certificate chain. Encrypted in TLS 1.3.
CertificateVerifyMessage 4. A signature over the handshake proving the server holds the cert's private key.
FinishedMessages 5 (server) & 6 (client). A MAC over the handshake confirming nothing was tampered with.
MACMessage Authentication Code — a keyed checksum proving a message wasn't altered.
