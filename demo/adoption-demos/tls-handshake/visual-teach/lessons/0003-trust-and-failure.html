1The chain of trust

Your client trusts a few dozen root CAs baked into its trust store. But roots don't sign websites directly — they sign intermediate CAs, which sign the leaf certificate the server actually presents. The server sends the leaf and the intermediate(s); the client walks the chain upward until it reaches a root it already trusts.

Each certificate is signed by the one to its right. The client verifies leaf → intermediate → root; the chain is valid only if it ends at a root the client already trusts.

Why an extra layer? Roots are kept offline and precious — if a root key leaked, every device on earth would need updating. An intermediate can be revoked and replaced without touching the roots in your trust store. The leaf is cheap and short-lived; the root is rare and guarded.

2The four checks

When the Certificate message lands (message 3 of the handshake), the client runs four independent checks. Any one failing aborts the connection — that's the error you see.

Check The question Ties back to
Chain Does the signature chain build up to a trusted root? Authentication — a CA you trust vouched for this key.
Hostname Does the cert's name list include the host I asked for? Authentication — the right server, not just a valid one.
Validity Is "now" inside the cert's not-before / not-after window? Freshness — expired keys may be compromised.
Revocation Has the CA revoked this cert early (OCSP / CRL)? Authentication — pulled before its expiry date.

3The four common failures

Almost every TLS error you'll meet is one of these. Each row pairs the failed check with the message curl prints and a one-line fix.

Failure What broke Typical fix
Expired certificate Validity check — "now" is past not-after. Renew/reissue the cert (often a lapsed auto-renewal).
Self-signed Chain check — the leaf signed itself; no CA above it. Get a cert from a real CA, or trust it deliberately (dev only).
Hostname mismatch Hostname check — the name you used isn't in the cert. Reissue covering the right name, or connect by a listed name.
Unknown / untrusted CA Chain check — ends at a root your client doesn't trust. Install the missing intermediate (server) or add the internal root (client).
Watch out — "unable to get local issuer certificate" most often means the server forgot to send the intermediate, not that the CA is untrusted. Browsers sometimes paper over this by caching intermediates; curl won't, which is why a site can work in Chrome but fail in a script. Same root cause, different leniency.

Inspect the cert yourself — to see the dates and names a check reads, run openssl s_client -connect example.com:443 -servername example.com, or in the browser click the padlock → "Connection is secure" → "Certificate". The fields you'll see are defined in the glossary.

5Your diagnostic checklist

This is the applied skill — interleaving everything so far. Work a real TLS error top to bottom; the first check that fails is your answer. Progress is saved, so this stays usable as a real runbook.

  • Read the exact error string Run curl -v https://the-host/. Copy the literal message — "expired", "self-signed", "subject name", "local issuer" each point at a different check. Don't guess from the padlock color.
  • Did it get past ServerHello? If the handshake fails before the Certificate line (Lesson 2), it's a version/cipher mismatch, not a trust problem. A cert error means messages 1–3 arrived fine.
  • Check validity dates Is "now" inside not-before / not-after? An expired (or not-yet-valid) cert is the single most common cause — and often just a dead auto-renewal.
  • Check the hostname Does the cert's Subject Alternative Name list the host you typed? www. vs apex, or a wildcard that doesn't cover a sub-subdomain, is a frequent mismatch.
  • Walk the chain to a root Does it reach a trusted root? "Self-signed" = no CA at all. "Unable to get local issuer" = a missing intermediate (server fix) or a private/internal root (add it to the client trust store).
  • Decide: server-side or client-side Renew / reissue with the right name / install the intermediate are server fixes. Trusting an internal root is a client fix. Name the side before you touch anything.

One free-recall pass to seal it — answer aloud, then reveal.

Name the four checks a client runs on a certificate.
Chain (builds to a trusted root), Hostname (name matches), Validity (not expired / not yet valid), and Revocation (not pulled early). Any one failing aborts the connection.
Self-signed vs. untrusted-CA — what's the difference?
Both fail the chain check. Self-signed has no CA above the leaf at all — it signs itself. Untrusted CA does have a CA chain, but it ends at a root your client doesn't carry (an internal/private root, or a missing intermediate).
