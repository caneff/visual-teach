Trust and failure

What the client actually checks in that certificate — and how to read the error, and act on it, when one of those checks fails.

⏱ ~15 min Needs: Lessons 1–2 Intermediate Lesson 3 of 3

By the end of this lesson you'll be able to:

  • Trace the certificate chain from leaf to a trusted root.
  • Name the four checks the client runs on a certificate.
  • Map a real TLS error to the check that failed — and what to do.
Why this matters for your goal: This is the payoff. Every "certificate error" you've ever clicked past is one of these four checks failing. After this lesson you'll read the message and know which — and whether it's safe to proceed (usually: no).

1The chain of trust

In Lesson 2 the server sent a Certificate message. It's rarely one certificate — it's a chain. The server's own leaf certificate is signed by an intermediate CA, which is signed by a root CA that already sits in your trust store. Validation walks up the chain, checking each signature, until it reaches a root the client trusts.

Intermediate CA
Root CA · in trust store

Each certificate is signed by the next one up. Trust flows down from the root the client already has.

Why a chain, not one cert? Roots are precious — compromise one and every cert under it is suspect — so CAs keep roots offline and sign with intermediates day to day. The client trusts the root; the root vouches for the intermediate; the intermediate vouches for the leaf. Trust is transitive down the links.

2The four checks

For the client to accept the leaf, all four of these must pass. Each maps to a guarantee from Lesson 1 — together they deliver authentication.

1 · Chains to a trusted root Every signature validates up to a root in the trust store.
2 · Hostname matches The URL's host is in the cert's Subject Alternative Names.
3 · Within validity window Now is between notBefore and notAfter.
4 · Not revoked The CA hasn't invalidated it (via CRL or OCSP).

3The common failures

Every certificate error you'll meet is one of these four checks failing. Learn the symptom → cause → action mapping; the live test hosts are subdomains of badssl.com.

Error you see Check failed What it means What to do
Certificate has expired Validity Past notAfter — usually a missed renewal Renew the cert; verify the server clock
Self-signed certificate Chains to root Signed by its own key; no CA vouches for it Use a CA-issued cert; trust it only for local dev
Unable to get local issuer Chains to root Chain doesn't reach a trusted root (missing intermediate) Install the intermediate on the server
Hostname mismatch Hostname URL host isn't in the cert's names Use the covered name, or reissue with the right SAN
Certificate revoked Revocation CA invalidated it (key compromise, mis-issuance) Get a fresh cert; never bypass revocation

4See it for real

Two commands surface everything above. Point curl at a broken host to read the error in one line, then use openssl s_client to inspect the live chain and dates yourself — don't take a tool's word for it.

Reproduce a known failure 
curl -v https://expired.badssl.com/
curl says
* SSL certificate problem: certificate has expired
* Closing connection
curl: (60) SSL certificate problem: certificate has expired
Inspect the chain and dates yourself 
# -servername sends SNI so the right cert comes back;
# the dates line shows the validity window from check #3
openssl s_client -connect expired.badssl.com:443 \
  -servername expired.badssl.com 2>/dev/null \
  | openssl x509 -noout -subject -dates -issuer
"Proceed anyway" disables authentication. Clicking through a cert warning keeps the encryption but throws away the one guarantee that you're talking to the real server. On a network you don't fully control, that's exactly when an attacker would be in the middle. Fix the cause instead — the checklist below is how.

5Applied: diagnose any TLS error

This interleaves all three lessons into one routine you can run on a real broken connection. Your progress is saved in this browser — keep it open as a working checklist.

  • Reproduce on the command line Run curl -v https://HOST/. The SSL certificate problem: line names the failure in plain words — start there, not in the browser.
  • Locate where in the handshake it broke Cert errors fire on the server's Certificate message (Lesson 2, step 3). A failure here means the key exchange was fine — the problem is trust, not crypto.
  • Inspect the real cert Run the openssl s_client … x509 -dates -subject -issuer command above. Read the actual hostname, dates, and issuer rather than trusting the summary.
  • Map it to one of the four checks Dates wrong → validity. Name wrong → hostname. Issuer unknown or self → chains-to-root. CA killed it → revocation. Exactly one will match.
  • Act on the cause, not the symptom Apply the fix from the failure table. If it's not your server and the cert is genuinely broken, the right move is to stop — not to click through.

7Recall from memory

Answer aloud first — then check. Last retrieval of the course; make it count.

Name the four checks the client runs on a certificate.
1. Chains to a trusted root · 2. Hostname matches · 3. Within the validity window · 4. Not revoked. All four must pass.
What does a self-signed error tell you about the chain?
It never reaches a trusted root — the cert is signed by its own key, so no CA in your trust store vouches for it. It fails check #1.
Why is clicking "proceed anyway" risky, in TLS terms?
It keeps confidentiality but discards authentication — you may be encrypting straight to an impostor. The connection looks secure while pointing at the wrong server.
