TLS 1.3 · Lesson 2 of 3

The handshake, step by step

Four messages, one round trip, and one beautiful trick: both sides end up with the same secret key while only public values ever cross the wire.

⏱ ~15 min Needs: Lesson 1 complete Intermediate Lesson 2 of 3

By the end of this lesson you'll be able to:

  • Put the TLS 1.3 handshake messages in order, from memory.
  • Say what each message carries and what it proves.
  • Explain how both sides derive one key without sending it.
Why this matters for your goal: Most TLS errors fire during the handshake. Knowing the message order tells you exactly how far a failed connection got before it broke — the first diagnostic question you'll ask in Lesson 3.

1The whole flow, once

Here is the full TLS 1.3 handshake. Read it top to bottom — it's a single round trip: the client speaks, the server answers, and application data starts flowing. (TLS 1.2 needed two round trips here; 1.3 folds the key exchange into the first messages.)

ClientHello
ServerHello
Certificate
CertificateVerify
Finished ×2

Client → ; ← Server (Hello through Finished) ; Client → Finished. From ServerHello onward, everything is already encrypted.

2What each message carries

Two messages set up the keys (the Hellos); two prove identity and integrity (Certificate/Verify and Finished).

# Message From Carries Proves / does
1 ClientHello Client Versions, ciphers, random, SNI, key share Opens the talk; offers the client's half of the key exchange
2 ServerHello Server Chosen version & cipher, random, key share Fixes parameters; supplies the server's half — keys now derivable
3 Certificate Server The server's cert chain Lets the client check identity (Lesson 3)
4 CertificateVerify Server A signature over the transcript so far Proves the server holds the cert's private key — not just a copy of the cert
5 Finished Both A MAC over the entire transcript Proves nothing was tampered with and both derived the same keys
Why CertificateVerify exists. Anyone can copy a public certificate — it's public. Holding the matching private key is the real proof. CertificateVerify is the server signing the live conversation with that private key, so a copied cert alone gets an attacker nowhere.

3The trick: a shared key nobody sent

Look again at messages 1 and 2: each side sends a key share — a public value. Neither ever transmits a secret. Yet right after ServerHello, both can compute the same session key. That's an ephemeral Diffie–Hellman exchange.

Keep the keys a black box. All you need is this one property:

Client computes
my private
+ your public
= shared secret
Server computes
my private
+ your public
= shared secret
Eavesdropper has
both publics
neither private
= nothing usable

Mix your own private with the other's public and both arrive at the identical secret. A watcher sees only the two public shares and can't reproduce it.

The one idea of the whole course. The session key is never sent — it's derived independently on each side from one private and one public value. And because the private values are ephemeral (thrown away after the connection), stealing the server's long-term key later still can't decrypt today's traffic. That property is called forward secrecy.

5Recall from memory

Say it before you open it — this is the retrieval that builds lasting memory.

List the handshake messages in order, with who sends each.
1. ClientHello (client) → 2. ServerHello (server) → 3. Certificate (server) → 4. CertificateVerify (server) → 5. Finished (server, then client).
How do both sides get the same key without sending it?
Each sends a public key share. Each combines its own private value with the other's public share and arrives at the identical shared secret. A watcher has both publics but neither private, so it can't reproduce the secret.
Why isn't presenting the certificate enough on its own?
Certificates are public and copyable. CertificateVerify makes the server sign the live transcript with the certificate's private key, proving it actually owns the key — not just a copy of the cert.

✦ What you earned

The TLS 1.3 handshake is ClientHello → ServerHello → Certificate → CertificateVerify → Finished, in one round trip. The Hellos carry public key shares that let both sides derive the same session key without ever sending it; Certificate/Verify and Finished prove identity and integrity.
