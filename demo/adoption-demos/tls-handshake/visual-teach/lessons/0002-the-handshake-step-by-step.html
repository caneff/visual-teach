TLS Handshake · Lesson 2

The handshake, step by step

Six messages turn an open wire into a private channel — and the shared key is never one of them. Here's the whole TLS 1.3 flow.

⏱ ~15 min Needs: Lesson 1 complete Intermediate Lesson 2 of 3

By the end of this lesson you'll be able to:

  • Put the six TLS 1.3 handshake messages in order and say what each carries.
  • Explain how both sides derive the same key without ever sending it.
  • Read the handshake lines in curl -v output and know what you're seeing.
Why this matters for your goal: When a connection fails, it fails at a specific message. Knowing the sequence is what lets you say "it got past ServerHello but choked on the certificate" instead of "TLS is broken." This is the backbone of the debugging you'll do in Lesson 3.

1The one hard idea: a key nobody sends

Here's the puzzle. The client and server have never met, the whole wire is public, and yet they need a shared secret key to encrypt everything. They can't just send the key — an eavesdropper would copy it. TLS 1.3 solves this with an ephemeral Diffie–Hellman exchange. Treat it as a black box with one magic property:

The paint-mixing trick — Both sides agree on a public base color. Each privately picks a secret color and mixes it with the base, then swaps the mixtures over the open wire. Each adds its own secret to the mixture it received — and both arrive at the same final color. An eavesdropper saw both mixtures but can't "un-mix" them to recover either secret. The final color is the shared key.

In the handshake, each side's "mixture" is called a key share — a public value carried in its Hello message. Each side combines the other's key share with its own private value to compute the identical session key. The session key itself never crosses the wire, and because the secret values are ephemeral (thrown away after), recording today's traffic doesn't help an attacker who steals the server's key tomorrow.

2The six messages

A full TLS 1.3 handshake is one round trip: the client speaks, the server replies with a whole flight at once, the client confirms. Read each message and what it carries:

ClientHello
ServerHello
Certificate
CertificateVerify
Finished (server)
Finished (client)

Messages 2–5 travel together in the server's single reply flight; from ServerHello onward everything is already encrypted.

# Message Direction What it carries
1 ClientHello Client → Server Offered TLS versions & cipher suites + the client's key share.
2 ServerHello Server → Client The chosen version & cipher + the server's key share. Now both can derive the key.
3 Certificate Server → Client The server's certificate chain (leaf → intermediate). Encrypted.
4 CertificateVerify Server → Client A signature over the handshake, proving the server holds the certificate's private key.
5 Finished Server → Client A MAC over everything so far — proof the handshake wasn't tampered with.
6 Finished Client → Server The client's matching MAC. Handshake done — application data flows.
Why the order is forced — Each step depends on the one before it. No key share (1 & 2) → nothing can be encrypted. No Certificate (3) → no public key to check the signature (4) against. No server Finished (5) → the client has nothing to confirm before it sends its own Finished (6). That dependency chain is exactly what the exercise below tests.

4See it for real

You don't need a packet sniffer — curl -v narrates the handshake. Run it and match the lines to the messages you just ordered:

curl -v https://example.com 
*  Trying 93.184.216.34:443...
* TLSv1.3 (OUT), TLS handshake, Client hello (1):     <- message 1
* TLSv1.3 (IN),  TLS handshake, Server hello (2):     <- message 2
* TLSv1.3 (IN),  TLS handshake, Encrypted extensions
* TLSv1.3 (IN),  TLS handshake, Certificate (11):     <- message 3
* TLSv1.3 (IN),  TLS handshake, CERT verify (15):     <- message 4
* TLSv1.3 (IN),  TLS handshake, Finished (20):        <- message 5
* TLSv1.3 (OUT), TLS handshake, Finished (20):        <- message 6
* SSL connection using TLSv1.3 / TLS_AES_256_GCM_SHA384
*  subject: CN=example.com
*  issuer: C=US; O=DigiCert Inc; CN=...
> GET / HTTP/2                                         <- now your request, encrypted
Try it now — Run curl -v https://example.com in your terminal. Those OUT/IN lines are the six messages. The cipher and issuer lines at the end are your preview of Lesson 3.

5Lock it in

Produce the answer aloud before revealing — that effort is the point.

The session key is never sent. So how do both sides have it?
Each side sends a public key share (in its Hello) and keeps a secret value. Each combines the other's key share with its own secret to compute the same session key. The key is derived independently on both ends, never transmitted — and an eavesdropper who saw both shares still can't reconstruct it.
Why must Certificate come before CertificateVerify?
CertificateVerify is a signature you check using the public key inside the certificate. Without the certificate (message 3) first, the client has no public key to verify the signature (message 4) against.
