TLS error — debug without guessing

When the padlock breaks: run two commands, read three lines, map the symptom to the one failed check. Built from Lesson 3.

Step 0 — ask the server to show its work

A
Quick surface — what failed, in one line: 
curl -v https://HOST/        # read the "SSL certificate problem:" line
B
Full inspection — chain, names, dates, issuer: 
openssl s_client -connect HOST:443 -servername HOST </dev/null 2>/dev/null \
  | openssl x509 -noout -subject -issuer -dates -ext subjectAltName
Read: subject (who the cert is for) · issuer (the CA that signed it) · notBefore/notAfter (validity) · subjectAltName (valid hostnames).

Step 1 — the four checks (every error is one of these)

Walk them top to bottom; the first that fails is your answer.

#CheckTypical messageUsual causeVerdict
1Chains to a trusted root self-signed certificate
unable to get local issuer certificate		 Self-signed host; private/corp CA; missing intermediate in server's chain usually config
2Hostname matches (SAN) certificate is not valid for NAME Wrong/placeholder cert; www vs apex; SNI not sent usually config
3Within validity window certificate has expired
not yet valid		 Lapsed/old cert served — or your own clock is wrong config / clock
4Not revoked certificate revoked CA pulled it early (key leak, mis-issue) take seriously

Step 2 — fast triage reflexes

Three tells that save you every time
  • Every host fails "expired" at once → it's your clock, not the web. Fix system time.
  • Browser works, curl/app fails with "local issuer" → server forgot to send the intermediate. Browsers cache it; bare clients don't. Server-config fix.
  • Connection still encrypts but warns (self-signed, unknown CA) → crypto fine, identity unvouched. Safe only if you independently trust that endpoint (e.g. your own dev box).

Step 3 — should I "proceed anyway"?

✓
Defensible on a host you control or expect to be self-signed/private-CA (internal tools, localhost, a dev server). You're supplying the trust the CA normally would.
✗
Don't on the public internet, on anything with a login or payment, or on a revoked cert. Hostname mismatch + a site asking for credentials is the classic phishing/MITM shape — stop.

Remember the half the checklist can't see. All four checks can pass on a copied certificate. The handshake's CertificateVerify step — signing the live exchange with the private key — is what stops a thief reusing a valid public cert. Chain validation says "the cert is legit"; CertificateVerify says "and this party truly holds it."
