TLS 1.3 · Reference

Glossary

The course nomenclature in one place. Every lesson uses these terms in exactly this sense. Keep it open while you read.

AEAD out of scope
Authenticated Encryption with Associated Data — the cipher style TLS 1.3 uses to get confidentiality and integrity in one step. Named here only so you recognise it; the internals are out of scope for this mission.
Authentication guarantee
Proof that you're talking to the party you intended, not an impostor. On the web it's one-sided: the server proves itself to the client. The bulk of the handshake and nearly every TLS error is about this. See Lesson 1.
Certificate (leaf) object
A public key bound to a hostname, signed by a CA. The "leaf" is the server's own certificate, at the bottom of the chain. Carries the public key, valid hostname(s), a validity window, and the issuer's signature.
Certificate Authority (CA) actor
A third party trusted to vouch for certificates by signing them after verifying domain control. Does its work ahead of time; it is not online during your handshake. Let's Encrypt is the largest.
CertificateVerify message
A handshake message in which the server signs the conversation so far with its certificate's private key. Proves it actually holds the cert, defeating anyone who merely copied the public certificate.
Chain of trust concept
leaf ← intermediate ← root. Validation walks up each signature until it reaches a root in the client's trust store. A broken or missing link = untrusted. See Lesson 3.
ClientHello message
The client's opening message: offered TLS versions, a cipher list, a random nonce, and its Diffie–Hellman key share. Starts key agreement immediately.
Confidentiality guarantee
Eavesdroppers see only ciphertext. Also called encryption or privacy.
Diffie–Hellman (ephemeral) — (EC)DHE black box
The key-agreement method. Each side sends a public share; each mixes its own private value with the other's public share to compute the same secret, which never crosses the wire. Ephemeral = fresh, throwaway keys per connection, giving forward secrecy. Treated as a black box; no math required.
Finished message
Sent by each side: a MAC over the entire handshake transcript. If anything was tampered with, or the sides disagree on what was said, the check fails and the connection aborts. This is integrity made concrete.
Forward secrecy property
Because key-exchange keys are ephemeral, recording today's traffic and stealing the server's long-term key later still won't decrypt it.
Hostname mismatch failure
The cert is trusted and in date but doesn't list the name you requested (checked against the SAN field). The key↔hostname binding doesn't hold.
Integrity guarantee
Tampering in transit is detected, not silently delivered. Enforced by the Finished MAC and by AEAD on application data.
Intermediate CA object
A CA certificate signed by a root and used to sign leaf certs, so the root's key can stay offline. The server must send it; forgetting to is the classic "unable to get local issuer certificate" bug.
Key share field
The key_share extension carrying one side's public DH value. One rides in ClientHello, one in ServerHello.
Private key / Public key black box
A keypair where one key undoes the other: what one locks, only the other opens; what one signs, only the other verifies. The server keeps the private key secret and never transmits it. The only property TLS needs from public-key crypto.
Revocation (CRL / OCSP) check
A CA's way to recall a certificate before its expiry (e.g. after a key leak). The client's fourth validity check.
Root CA trust anchor
A self-signed CA certificate shipped in the client's trust store and trusted a priori. The top of every valid chain.
Self-signed certificate failure
A cert that signed itself; the chain never reaches a trusted root. Crypto is fine, identity is unvouched. Routine on internal/dev hosts, a red flag on the public web.
ServerHello message
The server's reply: the chosen version and cipher, a random nonce, and its key share. After it, both sides can derive the session key and everything following is encrypted.
Session key secret
The symmetric key both sides derive from the DH exchange and use to encrypt the session. Never transmitted; independently computed on each end.
SNI (Server Name Indication) field
The hostname the client states in ClientHello so a server hosting many sites knows which certificate to present. (In openssl s_client, the -servername flag.)
Trust store object
The client's built-in set of root CA certificates it trusts a priori. Shipped by the OS or browser.
TLS 1.2 vs 1.3 contrast
1.3 cut the handshake from two round trips to one, encrypts the certificate, and removed legacy/insecure options. This course targets 1.3.
