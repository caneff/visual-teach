TLS debugging checklist

A TLS connection failed. Don't guess — walk these four checks in order.

The compressed essence of Lesson 3, built to print and pin up. Every common TLS error is one of four client-side checks failing. Find the failing check, and you have the cause.

First: get the facts

# curl — prints the verify reason and the cert it received
curl -v https://HOST

# openssl — shows the full chain, dates, names, and verify result
openssl s_client -connect HOST:443 -servername HOST </dev/null

Both tools name the failing check in plain words. Read that line first — it usually ends the investigation.

The four checks

1Signature chains to a trusted root

Does every signature validate, leaf → intermediate → root, ending at a root in your trust store? Watch for a missing intermediate — it looks like "untrusted issuer" but the server simply didn't send the full chain.

openssl: Verify return code: 0 (ok) means the chain is good. 19 (self-signed certificate in chain) or 20/21 = chain problem.

2Hostname matches

Does the cert's subject / Subject Alternative Name (SAN) list cover the exact host you requested? A trusted cert for the wrong name still fails.

curl: SSL: no alternative certificate subject name matches target host name.

3Not expired

Is "now" between the cert's notBefore and notAfter? Check your own clock too — a wrong local time fakes an expiry error on perfectly good certs.

openssl: read the Not Before / Not After dates in the cert.

4Not revoked

Has the CA declared the cert dead since issuing it (key compromise, mis-issuance)? Checked via CRL or OCSP. Rare, but treat a revoked cert as hostile.

Error → cause map

Message (curl / browser)Failed checkLikely cause
certificate has expired#3 validityRenewal didn't run — or your local clock is wrong.

Practice live

Trigger each failure on purpose, then predict the failing check before reading the warning, at badssl.com: expired, self-signed, wrong.host, untrusted-root, revoked .badssl.com.

One-line recall Chain, name, time, revocation. Four checks; any one failing aborts the handshake. Encryption can still succeed while authentication fails — that's precisely what a certificate warning is telling you.
