The TLS Handshake · Reference

Glossary

The course's shared vocabulary. Every lesson uses these terms exactly as defined here.

Quick-reference definitions for Lesson 1, Lesson 2, and Lesson 3. Scoped to TLS 1.3; designed to print on one sheet.

TLS protocol
Transport Layer Security. The protocol that gives a TCP connection confidentiality, integrity, and authentication. The "S" in HTTPS. This course targets TLS 1.3 (RFC 8446).
Confidentiality guarantee
Nobody between client and server can read the bytes. Provided by encrypting with the shared session key.
Integrity guarantee
Nobody can alter the bytes in flight without detection.
Authentication guarantee
You are talking to the real owner of the hostname, not an impostor. Established by the certificate plus the server's Finished message.
Client actor
The party that opens the connection (browser, curl) and decides which CAs to trust.
Server actor
The party that must prove it owns the requested hostname, by presenting a certificate and proving it holds the matching private key.
Certificate Authority (CA) actor
A trusted third party that signs certificates after verifying domain control. Clients trust a set of root CAs in advance.
Certificate object
A public key bound to an identity (hostname), signed by a CA. The object that makes authentication possible.
Keypair crypto
Two keys where one undoes the other. Treated as a black box in this course — we use its properties, never its math. One half is public, one private.
Public key / Private key crypto
The two halves of a keypair. The public half can be shared freely; the private half is never sent.
ClientHello message
First handshake message. Offers supported TLS versions and cipher suites, and includes the client's Diffie-Hellman key share.
ServerHello message
The server's reply. Selects the version and cipher and includes the server's key share. In TLS 1.3 it is bundled with the Certificate and Finished.
Finished message
Sent by each side over the whole handshake transcript using the derived key. Proves key possession and that the transcript was not tampered with.
Key share handshake
The public half of an ephemeral Diffie-Hellman keypair, sent in Hello messages. Both sides combine their own private value with the peer's key share to derive the same secret.
Diffie-Hellman (ephemeral) handshake
The key-exchange method where both sides derive an identical session key from exchanged public values; the secret is never transmitted. "Ephemeral" keypairs are discarded after the session. TLS 1.3 requires it.
Session key handshake
The symmetric key both sides independently derive and then use to encrypt application data. Never sent over the wire.
Forward secrecy property
Because keys are ephemeral, a later leak of the server's long-term key does not decrypt past recorded sessions. Mandatory in TLS 1.3.
Round trip (1-RTT) handshake
One there-and-back message exchange. A full TLS 1.3 handshake completes in one round trip before encrypted data flows.
Chain of trust trust
The path leaf → intermediate → root. The client validates signatures up the chain until reaching a root it already trusts.
Leaf / Intermediate / Root trust
Leaf = the server's own cert. Intermediate = a CA cert that signed the leaf. Root = a CA pre-installed in the device's trust store; the anchor of trust.
Trust store trust
The set of root CA certificates shipped with the OS / browser that the client trusts before any connection.
Self-signed certificate failure
A cert signed by its own key, chaining to no trusted root. Encrypts fine but fails authentication (chain check). Common in dev/internal use.
Hostname mismatch failure
A valid, trusted cert whose name does not cover the host requested.
Revocation (CRL / OCSP) trust
Mechanisms by which a CA declares a still-unexpired certificate invalid (e.g. after key compromise). CRL = published list; OCSP = live status query.
Term still fuzzy? Ask your teacher to define any of these in the context of a real connection you've seen — definitions stick better when attached to something concrete.
