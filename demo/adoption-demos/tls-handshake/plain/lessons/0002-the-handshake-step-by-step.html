The handshake, step by step

Four messages, one shared key that was never sent.

Your win for this lesson: put the TLS 1.3 handshake messages in order, say what each one carries, and explain how both sides end up with the same secret key without ever transmitting it.

In Lesson 1 we set the goal: confidentiality, integrity, authentication, established with a client, a server, and a CA. Now we watch them do it. TLS 1.3 is deliberately lean — the full handshake is one round trip.1 Picture the connection as four messages.

The flow, end to end

That's it at altitude. Three handshake messages and you're encrypted. Now the two ideas that make it work.

Idea 1 — the key share, and a secret never sent

The deep magic is in those "key shares." TLS 1.3 uses an ephemeral Diffie-Hellman exchange. Treat it as a black box with one remarkable property:2

The Diffie-Hellman property Each side generates a temporary keypair and sends only the public half (its key share). Combine your private half with their public half, and you compute a number. The remarkable part: they combine their private half with your public half and compute the same number. An eavesdropper sees both public shares and still cannot derive it.

So the session key is never transmitted — it is independently derived on each end from one private value (kept secret) and one public value (sent in the clear). The wire only ever carries public shares. That single property is why an attacker recording the entire handshake still can't read the traffic.

"Ephemeral" means those keypairs are thrown away after the session. Even if the server's long-term private key later leaks, today's recorded traffic stays unreadable — a guarantee called forward secrecy. TLS 1.3 requires this; TLS 1.2 merely allowed it.3

Contrast · TLS 1.2 Old TLS often used RSA key transport: the client encrypted a secret with the server's public key and shipped it. Steal that key later and all past traffic unlocks. TLS 1.3 removed this entirely — every handshake is ephemeral Diffie-Hellman.3

Idea 2 — what proves identity

A shared key alone gives confidentiality and integrity, but not authentication — you'd have a private channel to someone. Two parts of the server's reply close that gap:

  • The Certificate — the key-bound-to-hostname object from Lesson 1. The client checks it (the subject of Lesson 3).
  • The Finished message — computed over the whole transcript using the derived key. It proves the server actually holds the private key matching its certificate, and that nobody tampered with the messages so far.2

When the client sends its own Finished, both sides have confirmed they derived the same key over an untampered transcript. The channel is now private, intact, and authenticated — all three guarantees, in one round trip.
