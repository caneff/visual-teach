TLS 1.3 · Lesson 2 of 3

The handshake, step by step

One round trip. A handful of messages. By the end you can draw the TLS 1.3 handshake from memory and explain the magic trick at its centre: both sides compute the same secret key without ever sending it.

In Lesson 1 we said authentication is the hard part and the server's private key never goes on the wire. This lesson shows the choreography that delivers both. We target TLS 1.3, which trimmed the older 1.2 handshake from two round trips to one.1

The puzzle: a shared key over an open wire

The client and server have never met and share no secret. An eavesdropper sees every byte. Yet they must end up holding the same symmetric key to encrypt the session. How?

Ephemeral Diffie–Hellman, as a black box Each side invents a fresh, throwaway keypair and sends only its public share. There's a mixing function with one magic property: your private value mixed with their public value gives the exact same result as their private value mixed with your public value. Both sides compute it; the eavesdropper, holding only the two public shares, cannot.2

The secret is never transmitted — it's independently derived on both ends from public shares. "Ephemeral" means the keypairs are discarded after the handshake, so even if the server's long-term key later leaks, today's recorded traffic stays unreadable. You don't need the math; you need that shape. (One key share goes in each key_share extension.)

Why each message earns its place

ClientHello
"Here's what I support, and here's my half of the key." Offers versions and ciphers and already throws in a key share, so no extra round trip is needed to start key agreement.
ServerHello
"I choose these, and here's my half." With both halves on the table, each side runs the DH mixing function and gets the shared secret. Keys exist now.
Certificate + CertificateVerify
The split that matters: the Certificate claims an identity (Lesson 1's bound public key); CertificateVerify proves it, by signing the live handshake with the matching private key. A replayed certificate is useless without the private key to sign with.
Finished (both)
Each side hashes the entire conversation and MACs it. If a byte was altered in flight — or the two sides disagree on what was said — the MACs don't match and the connection aborts. This is integrity, made concrete.

Map it back to the three promises (Lesson 1): the key shares buy confidentiality, CertificateVerify buys authentication, and Finished buys integrity. The whole handshake is just those three promises being delivered, in order.

Practice — order the handshake

This is the core skill of the lesson. Put the messages in the order they travel, then check. Items in red are out of place — use the hint and retry until it's green. Do it twice from a fresh shuffle and it'll stick.

Sequence — drag-free; use ▲ ▼

  1. Certificate — server sends its bound public key
  2. ClientHello — versions, ciphers, client key_share
  3. Finished (server) — MAC over the transcript
  4. ServerHello — chosen params, server key_share
  5. Finished (client) — matching MAC, then app data
  6. CertificateVerify — sign the handshake with the private key

Hint: both Hellos first (you can't encrypt without keys), then the server claims then proves identity, then both sides Finish. Claim before proof; server finishes before client.

Question — the key idea

An attacker records the entire handshake in plaintext. Why can they still not derive the session key?

Right. The shared secret needs one side's private value mixed with the other's public share. The wire carries only the two public shares, so a recorder is stuck. Ephemeral keys mean even a future server-key leak won't help them.

Question — claim vs proof

A server presents a real, valid certificate for bank.com that it copied from the genuine site. Which message exposes the fraud?

CertificateVerify. Anyone can copy a public certificate. Proving identity requires signing the live handshake with the matching private key — which the impostor doesn't have. Claim is cheap; proof is not.
Free recall: in one sentence, how do two strangers end up with the same key without sending it?
Each sends a public key share; each mixes its own private value with the other's public share; the mixing function returns the same secret on both sides, and the wire never carried it.
