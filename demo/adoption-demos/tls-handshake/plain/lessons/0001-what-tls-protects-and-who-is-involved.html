TLS 1.3 · Lesson 1 of 3

What TLS protects, and who's involved

Before any bytes or key math: the smallest true picture of TLS — three promises, three actors, and one object (the certificate) that ties them together.

You hit this protocol thousands of times a day. The s in https is TLS — Transport Layer Security — sitting between HTTP and the raw network. HTTP still writes the request; TLS wraps the pipe it travels through. Your goal for this course is to debug a real TLS error without guessing. That starts here, with a model so simple it fits in one breath. We add mechanics in Lesson 2 and failures in Lesson 3.

The three promises

When a TLS connection is established, it secures the link in exactly three ways. Every TLS feature exists to deliver one of these three1:

Confidentiality (also: encryption / privacy)
An eavesdropper on the wire sees only ciphertext. They cannot read your request — not the URL path, not the cookies, not the body.
Integrity
An attacker in the middle cannot quietly alter the bytes. Any tampering is detected and the connection breaks, rather than delivering forged data.
Authentication
You can prove you're really talking to example.com and not an impostor. On the web this is almost always one-sided: the server proves who it is to the client; the client usually does not.
The trap Encryption alone is worthless without authentication. A perfectly encrypted conversation with the wrong party is exactly what an attacker wants. This is why most of the handshake — and almost every error you'll ever see — is about authentication, not encryption.

The three actors

Only three parties matter for the web case2:

ActorRoleWhat it holds
Client Your browser or curl. Starts the connection and decides whether to trust what the server presents. A built-in list of root CAs it already trusts (the trust store).
Server The website. Must prove it owns the hostname you asked for. Its private key + a certificate to present.
Certificate Authority (CA) A third party both sides defer to. Vouches for the server by signing its certificate after checking it controls the domain. Its own key, used to sign certificates. (Let's Encrypt is the largest.)

The CA is not online during your connection. It did its job earlier, when it signed the server's certificate. At handshake time the client just checks that signature against the roots it already trusts. That offline, ahead-of-time vouching is the whole trick of web identity.

What a certificate actually is

Strip away the file formats and a certificate is one sentence, made unforgeable:

Definition A certificate is a public key bound to an identity (a hostname like example.com), signed by a CA the client already trusts3.

Three pieces, and you'll meet all three again as failure modes in Lesson 3:

  • The public key — the server's half of a keypair. We treat the keypair as a black box: two keys where one undoes the other. What one key locks, only the other opens; a signature one key makes, only the other verifies. That single property is all TLS needs — we never touch the math.3
  • The identity — the hostname(s) the cert is valid for, plus a validity window (not-before / not-after dates).
  • The CA's signature — proof that a trusted authority checked the binding and stands behind it. Forge the public key or the hostname and the signature no longer matches.

So the server holds the matching private key but never sends it. It proves ownership by using the private key during the handshake; the certificate carries only the public half. Keeping the secret secret while still proving you hold it is the puzzle Lesson 2 solves.

Self-signed, foreshadowed. If a server signs its own certificate instead of getting a CA to sign it, the binding is still stated — but no one you trust vouches for it. The client has no path from its trusted roots to that signature, so it warns. Now you know why a self-signed cert is "valid crypto, untrusted source," not "broken." We'll diagnose it for real in Lesson 3.

Question 1 — the promises

A passive eavesdropper records your whole TLS session but cannot read a single request. Which guarantee stopped them?

Confidentiality. Encryption makes the bytes unreadable to anyone on the wire. Integrity would catch a change; authentication proves who you're talking to.

Question 2 — the certificate

A certificate binds a public key to a hostname. What makes that binding trustworthy to your browser?

The CA's signature. Trust comes from a signature that chains to a root your browser already ships with. A secret private key matters too, but it isn't what makes the binding believable to a stranger.

Question 3 — the trap

Why isn't encryption alone enough for a safe web connection?

Right. Without authentication you might have a flawlessly encrypted conversation with an attacker. That's why most of TLS — and most TLS errors — are about proving identity.
