Trust and failure

How a certificate is checked — and what each error really means.

Your win for this lesson: when a browser or curl throws a TLS error, name the likely cause and check it deliberately, using a four-point checklist instead of guessing.

Lesson 2 left one promise unkept: the client "checks the certificate." This lesson is that check — and every common TLS error is just one of these checks failing. This is where the whole course pays off.

The chain of trust

Your machine doesn't trust bank.com's certificate directly. It trusts a small set of root CAs, shipped with your OS and browser. Real sites aren't signed by roots directly; they're signed by an intermediate, which the root signed. The client follows the links upward — leaf → intermediate → root — until it reaches a root it already trusts.1

Leaf · the server's cert
bank.com — public key + hostname
▲ signed by
Intermediate CA
e.g. "Example CA R3"
▲ signed by
Root CA · pre-trusted on your device
in the OS / browser trust store

"Alice trusts Bob, Bob trusts Charlie, so Alice trusts Charlie."1 The root never goes on the wire — it's already on your machine. The server sends the leaf and intermediate; the client walks up to a root it holds.

What the client checks

For the connection to be authenticated, all four must hold. Any one failing aborts the handshake:

CheckThe question it answers
1. Signature chains to a trusted rootDoes each signature validate, up to a root in my trust store?
2. Hostname matchesDoes the cert's name cover the host I typed in the URL?
3. Not expiredIs "now" within the cert's validity window?
4. Not revokedHas the CA since declared this cert dead (CRL / OCSP)?2

The common failures — each is one check failing

Error you seeWhich check failedWhat it means
Certificate expired#3 validityValid once; the clock passed its "not after" date. Often a renewal that didn't run.
Self-signed certificate#1 chainSigned by itself, chains to no trusted root. Common on internal/dev servers.
Hostname / name mismatch#2 hostnameReal, trusted cert — for a different name than you requested.
Unknown / untrusted issuer#1 chainSigned by a CA not in your trust store (or a missing intermediate breaks the chain).
Certificate revoked#4 revocationThe CA pulled it — key compromise or mis-issuance. Treat as hostile.
Why a self-signed cert warns It's the question from Lesson 1, now answerable: a self-signed cert encrypts fine, but no CA vouched for it, so it fails check #1. You get confidentiality without authentication — a private line to an unverified party. That's exactly the gap the warning exists to flag.

Applied: the four-point diagnostic

Put it together. When a TLS connection fails, don't guess — walk the checklist. Reach for the TLS debugging checklist (a printable reference) and inspect with the tools you already know: 

# See the handshake, the cert chain, and the exact verify error:
openssl s_client -connect example.com:443 -servername example.com

# Or, more familiar — curl prints the reason and the cert it got:
curl -v https://example.com
  1. Read the verify error first. Both tools name the failing check ("certificate has expired", "self-signed certificate", "Hostname … doesn't match"). That word maps straight to the table above.
  2. Check expiry — the notBefore/notAfter dates, against the real clock (a wrong local clock fakes an expiry error).
  3. Check the name — does the cert's subject / SAN list cover the exact host you requested?
  4. Check the chain — does it reach a trusted root, with the intermediate present? A missing intermediate looks like "untrusted issuer."
Practice on purpose badssl.com hosts each failure live. Open expired.badssl.com, self-signed.badssl.com, wrong.host.badssl.com, and untrusted-root.badssl.com, and — before reading the warning — predict which check it breaks. Then confirm with curl -v. That prediction-then-check is the whole skill.
