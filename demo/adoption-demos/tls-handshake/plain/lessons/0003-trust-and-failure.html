TLS 1.3 · Lesson 3 of 3

Trust and failure

The payoff. The handshake from Lesson 2 only helps if the client trusts the certificate. Here's exactly what it checks, what each common error means, and a checklist to diagnose a real one without guessing.

The chain of trust

Your browser ships with maybe a few hundred root CA certificates in its trust store. The server's certificate is almost never signed by a root directly. Instead there's a chain:

The chain leaf (the server's cert, for example.com)  ←signed by←  intermediate CA  ←signed by←  root CA (in your trust store).

Validation walks up: each certificate's signature is checked using the public key of the one above it, until the chain lands on a root the client already trusts. The leaf trusts the intermediate; the intermediate trusts the root; the root is trusted a priori. No link in that chain may be missing or broken.1 Roots sign intermediates rather than leaves directly so the precious root key can stay offline, and a compromised intermediate can be revoked without reissuing every certificate on earth.

What the client checks

Even with a complete chain, the client runs four checks on the leaf before it trusts the connection. Every common TLS error is one of these four failing2:

CheckThe questionIf it fails
Chains to a trusted root Does each signature validate up to a root in my store? Unknown / untrusted CA, or self-signed
Hostname matches Is the name I typed listed in the cert's SAN field? Hostname mismatch (wrong.host)
Within validity window Is now between not-before and not-after? Expired (or not-yet-valid)
Not revoked Has the CA recalled this cert early (via CRL / OCSP)? Revoked

The four failures, decoded

Expired certificate
Valid chain, valid hostname — but the clock is past not-after (or the cert is renewed and the server is still serving the old one). Also fires if your own machine's clock is wrong. Check the date before you blame the server.
Self-signed certificate
The cert signed itself; the chain never reaches a trusted root. The crypto is fine and the connection would be encrypted — but nobody you trust vouches for the identity. Common and often harmless on internal/dev hosts; a red flag on the public web. (Foreshadowed back in Lesson 1.)
Hostname mismatch
The chain is trusted and the cert is in date — but it's for the wrong name (you asked for www.site.com, the cert says only site.com; or it's a default/placeholder cert). The binding from Lesson 1 — key ↔ this hostname — doesn't hold.
Unknown / untrusted CA
Signed by a CA your client doesn't have in its store: a private corporate CA, or a missing intermediate the server forgot to send (so the chain can't be completed). Very often a server-config bug, not an attack.

Why CertificateVerify still matters here. All four checks pass on a copied certificate too. What a copy can't do is sign the live handshake — the CertificateVerify step from Lesson 2. Chain validation proves the cert is legit; CertificateVerify proves the party in front of you actually holds it. You need both.

Applied — diagnose it without guessing

This interleaves all three lessons into one move you'll actually make. When a connection fails, ask the server to show its work:

# One line that surfaces almost every failure mode:
curl -v https://expired.badssl.com/

# The full inspector — chain, names, dates, issuer:
openssl s_client -connect wrong.host.badssl.com:443 -servername wrong.host.badssl.com

# In curl's output, read these lines:
#   *  SSL certificate problem: certificate has expired      <- which check failed
#   *  subject: CN=*.badssl.com                              <- the hostname binding
#   *  issuer:  CN=...                                       <- who signed it (the CA)

badssl.com runs a subdomain for each failure on purpose3 — a safe range to practise on. Map the symptom to the failed check:

Diagnosis 1

curl https://self-signed.badssl.com/ → "self-signed certificate." Which of the four checks failed?

Chain to a trusted root. A self-signed cert never reaches a root in your store. The connection would still encrypt — it just can't prove whose it is.

Diagnosis 2

A monitoring job that worked yesterday now reports "certificate has expired" on every HTTPS host at once. Most likely cause?

Your clock. "Within the validity window" is checked against local time. Every host failing at once is the classic fingerprint of a wrong system clock — fix the clock, not the certs.

Diagnosis 3

A browser reaches a site fine, but curl on your server says "unable to get local issuer certificate." Best first guess?

Missing intermediate. Browsers often cache intermediates and paper over the gap; a bare client like curl can't, so the chain won't complete. A server-config bug, not an attack — fix the chain the server sends.
