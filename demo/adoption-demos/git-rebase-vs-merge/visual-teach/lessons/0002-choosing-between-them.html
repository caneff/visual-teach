Lesson 2 of 3 · Git Merge vs Rebase

Choosing Between Them

~15 minutes · ← The Two Shapes of History · Rebase Without Fear →

Now that you can picture what each command does to the commit graph, the question becomes: which one do you reach for? There is a single deciding question that covers the vast majority of real situations.

The deciding question

Has this branch been pushed to a remote, and might a teammate have pulled it? 

Is the branch pushed AND might someone else have pulled it?
│
├─ YES — it is a shared branch
│   └─ Use git merge to bring in new commits
│
└─ NO — it is your private branch alone
    └─ git rebase is safe and keeps history clean

That is the whole rule. Everything else is a secondary preference. Work through the scenarios below until it feels automatic.

Why does "shared" change everything? When you rebase, the commits get new SHA hashes. If a teammate already has the old hashes, their git now has commits that do not match yours — two diverging versions of the same branch. You have to force-push to override them, and they have to do gymnastics to reconcile. That is the accident you are avoiding.

Reading the rule for common situations

Scenario A — Tidying a local feature branch before opening a PR

Your branch is local; you have not pushed it yet. Rebase it onto main to get a clean, linear chain that is easy for reviewers to read. 

git checkout feature/my-thing
git rebase main
Source: Atlassian — Merging vs. Rebasing

Scenario B — Updating a shared team branch with main's latest

Multiple people push to develop or a release/x branch. Someone else may have commits you do not have. Use merge — it adds one clearly-labelled integration commit and touches nothing existing. 

git checkout develop
git merge main

Scenario C — Bringing main's new commits into your open feature branch

Your branch is open as a PR (pushed) and a teammate may have pulled it to test. Merge main into your branch. Do not rebase — your branch is now shared. 

git checkout feature/my-thing
git merge main

Scenario D — A private backup-only push

You pushed your branch to remote purely as a backup, and you are certain no teammate has pulled it. You can still rebase — the branch is effectively private. Immediately after, you will need to force-push: git push --force-with-lease. The lease flag aborts if someone else pushed since your last fetch, which catches the rare case you were wrong about "private."

The PR merge itself is always a merge. Rebase is about how you update your branch while working. When you close the PR, that is a merge (or squash merge) on the main side — that choice belongs to the team's PR workflow, not to you alone.

Scenario 1. You have been building feature/dark-mode for two days. It is local — never pushed. Main has three new commits. You want to bring them in before finishing.

  • git merge main — add a merge commit into the feature branch
  • git rebase main — replay your commits onto the latest main

The branch is local and private. Rebasing is safe and preferred: your feature commits move cleanly on top of the latest main, giving reviewers a linear diff with no noise merge commit.

Scenario 2. Your team shares a develop branch. A critical security fix was merged into main this morning. You need that fix in develop right now.

  • git merge main — adds one integration commit onto the shared branch
  • git rebase main — rewrites every commit on the shared branch

develop is shared — your three teammates have commits on it you may not have. Rebasing would rewrite every existing commit on develop, creating a conflict nightmare for everyone else. Merge adds a single, clearly-labelled integration commit and touches nothing that already exists.

Scenario 3. You pushed your feature branch last week and shared the link with a teammate, who pulled it to test locally. Main has moved ahead by eight commits. You need them.

  • git merge main — safe because the branch is now shared
  • git rebase main — cleaner history, teammate can just re-pull

The moment your teammate pulled your branch, it became shared. Rebasing rewrites the hashes — your teammate's local copy now has diverging history. "They can just re-pull" only works if they have not committed on top of your branch. Merge is the safe, respectful choice.

Scenario 4. Your local-only feature/api-cache branch has four commits. Main just landed a bug fix you need to test against before finishing your work.

  • git merge main — pull in the fix with a merge commit
  • git rebase main — replay your four commits on top of the fix

Local and private. Rebase gives you a clean, linear history: your four commits sitting neatly on top of the latest main. When you open the PR the reviewer sees only your four commits — no merge noise. The merge option also works, but leaves a merge commit that adds no information to a private branch.

A quick summary you can use right now

Private branch: rebase to keep history clean.
Shared branch: merge to keep teammates safe.
Not sure which it is? Assume shared and merge.
