Git merge vs. rebase · Lesson 2

Choosing between them

You can draw both shapes. Now the real skill: given a situation in front of you, decide merge or rebase — and justify it in one sentence.

⏱ ~12 min Needs: Lesson 1 Beginner Lesson 2 of 3

By the end of this lesson you'll be able to:

  • Ask the one question that decides merge vs. rebase
  • Choose correctly for everyday scenarios and state why in a sentence
  • Recognise when "it depends" really means "ask my team"
Why this matters for your goal: the anxiety comes from having no rule — every integration feels like a gamble. One sharp question removes the gamble for the cases you hit daily.

1The one question that decides it

Almost every real decision collapses to a single question about the commits you're about to move:

Have these commits been pushed and shared with anyone else? If yes → merge (rewriting shared commits breaks everyone who has them). If no → you're free to rebase for a clean, linear history.

This is the Golden Rule of Rebasing, and it comes straight from the Pro Git book: never rebase commits that exist outside your repository and that people may have based work on. Get this one rule right and rebase stops being dangerous. We'll drill the recovery side of it in Lesson 3.

2The everyday cases at a glance

Three situations cover most of your week. Here's the default call for each — and the reason behind it, which matters more than the call:

Situation DefaultChoice Because
Tidy a local feature branch before opening a PR rebase The commits are private and unpushed — rewriting them is free, and reviewers get a clean line.
Update your branch with the latest main (branch is yours alone) rebase Replays your work on top of main; history stays linear and your changes sit clearly on the newest base.
Integrate main into a long-running shared branch merge Others have based work on that branch — rebasing it would rewrite commits they already hold. Merge is safe and records the real timeline.
Notice the pattern: rebase for private cleanup, merge for shared integration. "Linear and tidy" is the upside of rebase; "true, safe, and shared" is the upside of merge. Neither is morally better — they optimise for different things.

4Drill: make the call

Six situations below. For each, decide before you click — commit to an answer, then read the justification. Aim to predict the reason, not just the verdict. Your running score is at the top.

Decision drill 0 / 0 right
Local cleanup

You've got six messy "wip" commits on a feature branch that exists only on your laptop. You want a clean, current history before opening the PR. main has moved ahead since you branched.

the commits are private and unpushed, so rewriting them costs nothing. git rebase main replays them on the newest base and gives reviewers one clean line to read.
Shared branch

A long-running develop branch that five teammates pull from has fallen behind main. You need to bring main's changes into develop.

develop is shared — rebasing it would rewrite commits your teammates already hold, the exact thing the golden rule forbids. git merge main brings the changes in safely and records when it happened.
Update before PR

Your three-commit feature branch is pushed only to your own fork; no one else uses it. main moved ahead and you want the latest changes before you finish up.

pushed, but only to your own branch that nobody has based work on — still effectively private. git rebase main keeps history linear and lands your work cleanly on top of the latest.
Someone else has it

A colleague has already pulled your feature branch to test it locally. Now main has moved and you want your branch updated with it.

the moment someone else holds your commits, rebasing orphans their copy and forces a painful re-sync. Merge in main — or explicitly coordinate before any force-push.
Into shared main

Your feature is finished and reviewed. You want to combine it into the shared main branch that the whole team deploys from.

main is the canonical shared branch, so you never rebase it. Integrating the feature is a merge (this is exactly what a PR "merge" does). Tidy the feature with rebase first if you like — then merge it in.
Private spike

Two tiny commits on a throwaway spike branch on your machine. You realise they belong on top of a different base branch than the one you started from.

nothing is shared, so you're free to move the commits. Rebasing onto the right base is the cleanest way to relocate private work.
✦ Drill complete. If you predicted the reason each time — not just the verdict — you've got the decision skill. The reasons all trace back to one question: is it shared?
When the rule runs out: some teams mandate a house style — "always rebase for linear history" or "always merge, never rewrite." A team convention overrides these defaults. If you don't know yours, that's a great question for the r/git community or your tech lead — and worth recording so future-you doesn't re-litigate it.
