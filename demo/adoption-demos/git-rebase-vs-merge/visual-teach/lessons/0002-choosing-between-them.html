Merge vs Rebase · Lesson 2 of 3

Choosing between them

You can picture both shapes now. This lesson turns that picture into a decision you can make in two seconds, for any branch on your screen.

⏱ ~12 min Needs: Lesson 1 Beginner Lesson 2 of 3

By the end of this lesson you'll be able to:

  • Apply one question that decides merge vs rebase almost every time.
  • Pick correctly for three everyday scenarios — and justify it in a sentence.
  • Spot the one case where "pushed" doesn't mean "off-limits".
Why this matters for your goal: the anxiety isn't really about the commands — it's about not having a rule. Once a single question decides it, the coin-flip dread is gone and you just answer.

1The one question that decides it

Almost every merge-vs-rebase call collapses to a single question about the commits you'd be rewriting:

Has anyone else got these commits yet?
No (the commits are private — only on your machine, or on a branch nobody has based work on) → rebase is safe, and a clean straight line is the nicer result.
Yes (the commits are shared — others have pulled them or build on them) → merge. Never rewrite history other people already have.

That's the golden rule of rebasing, stated as a decision. We'll meet it again, with the recovery tools, in Lesson 3. Here's how the two stack up once you've answered that question:

  git merge git rebase
History shape Fork preserved, tied off by a merge commit One straight, linear line
Original commits Untouched — same hashes Replaced by new-hash copies
Safe on shared commits Always No — rewrites history
Records that branches met Yes — the merge commit is the record No — looks like it was always linear
Reach for it when… Integrating shared history, or you want the merge recorded Tidying or updating a private branch before review
Updating a private branch is the everyday rebase. To pull the latest main into your own feature branch without a noisy "Merge branch 'main'" commit, rebase instead of merge:
keep your branch current, linearly 
git fetch origin
git rebase origin/main      # replay your work on top of the latest main
# shortcut for the same idea when pulling:
git pull --rebase

2Decide it — three real scenarios

This is the lesson. Read each scenario, decide before you look at the options, then pick. Wrong picks explain themselves — keep going until it's right. The scenarios are deliberately mixed so you practice the question, not a pattern.

Scenario A — your laptop only

Your feature/login branch lives only on your laptop — never pushed. main has moved ahead and you want its latest changes before you finish.

Merge main in, or rebase onto it?

Scenario B — the shared release branch

release/2.0 is a long-running branch three teammates pull from and commit to daily. It needs the latest main.

Bring main in with merge, or with rebase?

Scenario C — pushed, but yours alone

You pushed feature/login to open a draft PR. CI ran on it, but nobody else has based any work on it. main moved on, and you'd like a tidy, linear branch before a reviewer looks.

What's the right move here?

Scenario C is the trap most people fall into — in both directions. Some treat every pushed branch as frozen (and never tidy anything); others rebase a branch teammates are building on and cause chaos. The dividing line is never "did I push?" — it's "does anyone else depend on these commits?"

✦ What you earned

One question now decides it: has anyone else got these commits? No → rebase (private cleanup, a tidy line). Yes → merge (never rewrite shared history). And you learned the edge case — "pushed" ≠ "shared", so a solo PR branch can still be rebased, with --force-with-lease.
