Lesson 1 of 3 · Git Merge vs Rebase

The Two Shapes of History

~12 minutes · Next: Choosing Between Them →

You have been merging everything into main and it works — but your git log --graph looks like a subway map. There is a second path: git rebase, which produces a completely different shape of history. Before you can choose between them confidently, you need to see exactly what each command does to the commit graph.

What is a commit graph?

Every git commit stores a pointer to its parent commit. That chain of pointers forms a directed acyclic graph — what git log --graph renders as branching lines. Most commits have one parent; a merge commit has two. The shape of that graph is what merge and rebase change in fundamentally different ways.

What git merge does

When you run git merge feature from main, git finds the nearest common ancestor of both branches, then creates a new merge commit with exactly two parent pointers — the tip of main and the tip of feature. The branching structure is recorded permanently in the graph. 

Before merge:

      A---B---C   ← feature
     /
D---E---F---G     ← main

After  git merge feature  (run from main):

      A---B---C
     /         \
D---E---F---G---H   ← main
                ^
                merge commit H has two parents: G and C

The key property: the graph is permanent evidence of what happened. Anyone reading the history can see that commits A, B, C came from a separate branch and rejoined main at H. The merge commit H carries two parent SHAs — that is what makes it a merge commit.

Source: Pro Git §3.2 — Basic Branching and Merging

What git rebase does

When you run git rebase main from the feature branch, git takes each commit on feature that is not on main and replays them one by one on top of the current tip of main. The result looks as though the feature work was always built on the latest main — a straight line. 

Before rebase:

      A---B---C   ← feature
     /
D---E---F---G     ← main

After  git rebase main  (run from feature):

                  A'--B'--C'  ← feature
                 /
D---E---F---G   ← main (unchanged)

The primes matter: A′, B′, C′ are new commits. They contain the same diff as A, B, C but they have different SHA hashes, because their parent is now G instead of E. The old commits A, B, C still exist in git's object store until garbage-collected, but the feature branch pointer now follows the new linear chain.

New hashes — always. Rebase never moves existing commits; it creates copies with a new parent. Because a commit's SHA is a hash of its content plus its parent SHA, any change of parent produces a new hash. This is why shared branches and rebase are dangerous — and it is the root of the golden rule you will learn in Lesson 3.
Source: Pro Git §3.6 — Rebasing

Side by side

git merge is additive: it appends a new merge commit that records the join, preserving the full branching shape. git rebase is reconstructive: it rebuilds your commits with a new starting point, erasing the branching shape and producing a line.

Property git merge git rebase
Commit hashes on feature Unchanged All change
History shape Branching graph Linear line
Records when branches joined Yes (merge commit) No
Creates a new commit Yes (merge commit) Yes (replayed copies)

After running git merge feature, how many parent commits does the resulting merge commit have?

  • One parent — only the previous tip of main
  • Two parents — the tip of main and the tip of feature
  • Three parents — the common ancestor plus both tips
  • Zero parents — merge commits are root commits

A merge commit records exactly two parent SHAs: the branch you were on (main) and the branch you merged in (feature). This dual-parent pointer is what lets git log --graph draw the branching lines backward through history. The common ancestor is computed internally but is not stored as a parent.

You rebase your feature branch. What is true of the replayed commits (A′, B′, C′) compared to the originals (A, B, C)?

  • They have identical SHA hashes — only the branch pointer moves
  • They are gone — rebase deletes the original commits
  • They have new SHA hashes — the same diff, a new parent
  • They gain an extra parent pointing back to the old base

A SHA is a hash of the commit's content plus its parent SHA. Because rebase changes the parent (from the old base E to the new base G), the hash changes — even when the diff is byte-for-byte identical. The originals A, B, C are not deleted; they just become unreachable from the branch pointer.

A coworker needs to audit exactly when two lines of work were combined. Which history gives them that answer?

  • A merged history — the merge commit records the exact join point
  • A rebased history — the linear chain is easier to read through
  • Both histories preserve the same information about the join
  • Neither history records when the two branches rejoined

Only a merge commit records the exact moment two histories converged. In a rebased history the feature commits appear to be direct descendants of main — the fact that they were developed on a separate branch is erased. That traceability trade-off is one of the real factors in the merge vs rebase decision.
