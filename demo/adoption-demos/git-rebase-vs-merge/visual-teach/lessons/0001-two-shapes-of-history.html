The two shapes of history

Before you can choose between merge and rebase, you need to see what each one does to your commit graph. That's the whole lesson: two pictures, burned into memory.

⏱ ~10 min Needs: basic add/commit/push Beginner Lesson 1 of 3

By the end of this lesson you'll be able to:

  • Read a commit graph: commits, parents, and branch labels.
  • Draw what git merge does — and spot a merge commit.
  • Draw what git rebase does — and explain "new hashes".
Why this matters for your goal: The fear of rebase ("it'll lose my work") comes from not picturing what it does. Once you can see both shapes, the command stops being scary and becomes a choice. No decisions yet — that's Lesson 2. Today is just seeing.

1A commit graph is just dots and arrows

A commit is a snapshot of your project plus a pointer to the commit that came before it — its parent. Chain them and you get a graph that grows left → right in time. A branch (like main) is nothing more than a movable label pointing at the newest commit on a line.[1]

Here's the situation every integration starts from. You branched feature off main at commit B and made two commits, C and D. Meanwhile a teammate landed E on main. The two lines have diverged:

on main on feature

B is the point where the branches split — the common ancestor. merge and rebase are just two different ways to bring these two lines back together.

2merge ties the lines together

Run git merge feature while on main and git creates one new commit — a merge commit — whose special trick is having two parents: the tip of each line it joined (here, E and D). Nothing else moves. Your original commits keep their exact hashes; the branching shape is preserved forever.[2]

on main on feature merge commit

M is the merge commit. Two parents = "two lines joined here." History stays honest: anyone can see the work happened on a branch and was integrated.

Fast-forward — if main hadn't moved (no E), there'd be nothing to reconcile. Git would just slide the main label forward to D — a fast-forward, no merge commit at all.

3rebase replays onto a new base

Run git rebase main from feature and git does something different: it takes your commits C and D, sets them aside, moves to the tip of main (E), and replays them there one at a time. The manual's own one-liner is "reapply commits on top of another base tip."[3]

Each replayed commit is a brand-new commit with a new hash (C′, D′) — same changes, different identity, because its parent changed. The originals are left behind (shown ghosted) and later garbage-collected. The result is one straight line:

on main replayed copy (new hash) original (abandoned)

C′ and D′ hold the same changes as C and D but are new commits. The history now reads as one clean line, as if you'd started from E all along.

This is the key to losing the fear. Rebase doesn't delete your work — it copies it to a new base. The changes are identical; only the commit hashes change. "Rewriting history" means "making new commits", not "throwing code away". Lesson 3 shows the undo button that proves it.

4Side by side

The same divergence, resolved two ways. This contrast is the whole course in one frame:

Merge = honest record, branching graph, extra merge commit. Rebase = clean line, new commit hashes, no merge commit. Neither is "right" — they're different shapes for different jobs. See the side-by-side table in the quick reference.

6Try it on a throwaway repo (optional, 2 min)

Seeing it on your own screen makes it real. Paste this into a scratch directory and read the graph it prints — no risk, nothing is pushed:

see both shapes locally 
mkdir /tmp/shapes && cd /tmp/shapes && git init
git commit --allow-empty -m A
git commit --allow-empty -m B
git switch -c feature
git commit --allow-empty -m C
git commit --allow-empty -m D
git switch main
git commit --allow-empty -m E
git log --oneline --graph --all     # the diverged shape from section 1

Now try git switch feature && git rebase main then git log --oneline --graph --all again — watch the fork collapse into a line. (How to do this safely on real branches is Lesson 3.)

I'm your teacher — ask me anything

This lesson is just the two pictures. If either one is still fuzzy, ask me to redraw it with your own branch names, or to walk a specific commit through a rebase step by step. A few openers:

Can you redraw the merge graph using my real branch names? Why does changing the parent change the hash? Where do the abandoned commits actually go?

✦ What you earned

You can now read a commit graph and draw both moves: merge joins two lines with a two-parent merge commit and keeps the fork; rebase replays your commits onto a new base as new hashes, making one straight line. No work is destroyed either way.
