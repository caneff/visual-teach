Merge vs Rebase · Lesson 1 of 3

The two shapes of history

Before you can choose between merge and rebase, you need to see exactly what each one does to your commit graph. That's all this lesson does — build the picture.

⏱ ~10 min Needs: basic add / commit / merge Beginner Lesson 1 of 3

By the end of this lesson you'll be able to:

  • Read a commit graph and point to the branch point.
  • Predict the shape a merge leaves behind (a fork that rejoins).
  • Predict the shape a rebase leaves behind (a straight line, new hashes).
Why this matters for your goal: the fear of rebase ("it'll lose my work") comes from not being able to picture what it does. Picture it once, accurately, and the dread has nowhere to live. Decisions come in Lesson 2; this is the foundation they stand on.

1One starting point, two integrations

Every merge-vs-rebase decision starts the same way. You branched off main to build something. While you worked, a teammate pushed to main. Now history has diverged — it forks at the commit you branched from:

You branched at b2d. You added 9f0 and 3ab. Meanwhile main moved on to e5f. The fork is real — both commands below are just two ways to bring these lines back together.

merge and rebase answer the same question — "combine these two lines of work" — but they leave the graph in two very different shapes. That shape is the whole story.

2Merge keeps the fork — and ties it off

A merge creates one brand-new commit — a merge commit — with two parents: the tip of main and the tip of your branch. Nothing that already existed is changed or moved. The fork stays visible forever; the merge commit is the knot that ties the two strands back together. [Pro Git 3.2]

on main already your feature work the new merge commit

m7e is the merge commit. Its two parents are e5f and 3ab. Every original commit keeps its exact hash — nothing was rewritten.

Run it, then read the graph it printed:

You run 
git checkout main
git merge feature/login
git log --graph
*   m7e1b2c (HEAD -> main) Merge branch 'feature/login'
|\
| * 3ab77d1 (feature/login) Add remember-me box
| * 9f0c4e2 Add login form
* | e5f8a90 Update CI matrix
|/
* b2d5c10 Tidy README
* a1c0f3d Initial commit
Read the ASCII graph like the picture above. The |\ is the fork being tied off; the |/ below is the branch point. The two * | / | * columns are the two strands running side by side. Same shape, drawn in text.

3Rebase moves your commits — and straightens the line

A rebase does something completely different. It takes your feature commits, sets them aside, fast-forwards your branch to the tip of main, and then replays your commits one by one on top. The result is a single straight line — as if you'd started your work from the latest main all along. [Atlassian]

on main already your work, replayed

No merge commit. No fork. Your two commits now sit after e5f — but they are copies: d4a′ and 7b1′ are new commits with new hashes. The originals (9f0, 3ab) are left behind, unreferenced.

You run 
git checkout feature/login
git rebase main
git log --graph
* 7b1e44a (HEAD -> feature/login) Add remember-me box
* d4a90c2 Add login form
* e5f8a90 (main) Update CI matrix
* b2d5c10 Tidy README
* a1c0f3d Initial commit
This is the fact that defuses the fear. "Rebase rewrites history" simply means new commits with new hashes replace the old ones. Your changes are not lost — they're copied forward. The old commits even stick around in the reflog for a while, which is your safety net (Lesson 3). Rewriting is only dangerous when someone else already has the old commits — that's the one rule, and it's all of Lesson 3.

4Recall — predict the shape

Don't re-read the diagrams. Answer from memory — getting it slightly wrong and correcting is what makes it stick.

You run git rebase main on your feature branch. What happens to your two feature commits?

✦ What you earned

Two shapes, locked in: merge keeps the fork and ties it off with a two-parent merge commit (nothing rewritten); rebase straightens history into one line by replaying your commits as new-hash copies. You can now predict the graph before you run the command.

I'm your teacher — ask me anything

Diagrams are easy to nod along to and hard to truly hold. If any part of the graph didn't click, ask me to redraw it with your own repo's branch names, or to walk a single commit through the rebase step by step. A few openers:

Where do the "lost" original commits actually go? Why does main move while I'm on my branch? Can you redraw this with three feature commits?
