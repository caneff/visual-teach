Lesson 3 of 3 · Git Merge vs Rebase

Rebase Without Fear

~15 minutes · ← Choosing Between Them · Reference: Cheat Sheet →

The fear of rebase usually comes down to two questions: "What if I lose my commits?" and "What if I break my teammates' work?" This lesson answers both, then gives you a concrete workflow so rebase feels routine.

The golden rule

The golden rule of rebasing

Never rebase commits that have been pushed to a shared branch — a branch that other people have or might have pulled.

That one constraint is the complete safety contract. If you honour it, rebase is safe. If you need a deeper rationale, it is this: when you rebase a pushed branch, the old commits and the new (rewritten) commits both exist in the git world simultaneously. Anyone who pulled the old ones now has a branch that diverges from yours. Resolving that requires either a force-push (which may overwrite their work) or a messy second merge. The golden rule prevents the situation from arising.

Source: Pro Git §3.6 — "The Perils of Rebasing"

The escape hatch: git reflog

Rebase does not delete commits. It creates new copies and moves the branch pointer. The original commits are still in git's object store — they are just unreachable from the branch pointer. git reflog logs every position HEAD has ever occupied, including the moment before your rebase started. 

# After a rebase you regret, run:
git reflog

# Output looks like:
# 4e3a1b2 HEAD@{0}  rebase (finish): returning to refs/heads/feature/my-thing
# 91d0fe3 HEAD@{1}  rebase (start): checkout main
# 91d0fe3 HEAD@{2}  commit: add caching layer       ← your branch BEFORE the rebase
# ...

# Reset to the state before the rebase:
git reset --hard HEAD@{2}   # use the index that matches your pre-rebase commit

The HEAD@{n} notation reads as "where HEAD was n moves ago." The reflog expires after 90 days by default, so this recovery path is available for any recent mistake.

Source: git-reflog official documentation

Resolving conflicts during a rebase

Unlike merge — which collects all conflicts into one commit — rebase applies commits one at a time, so you may need to resolve a conflict more than once if separate commits touched the same lines.

When git pauses on a conflict it tells you:

CONFLICT (content): Merge conflict in src/auth.js
error: could not apply 91d0fe3... add caching layer
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".

The resolution loop is:

  1. Open the conflicted file. Resolve the <<<<<<< … >>>>>>> markers manually.
  2. Save the file.
  3. git add <resolved-file> — mark it resolved. Do not run git commit.
  4. git rebase --continue — move to the next commit.
  5. Repeat from step 1 if another conflict appears.

To abort and return to your pre-rebase state at any point:

git rebase --abort

--abort is a clean undo — it restores the branch pointer to exactly where it was before you ran git rebase. No reflog needed.

Source: git-rebase official documentation

After the rebase

Use --force-with-lease, not --force. After rebasing a private branch you have already pushed, a regular git push --force will blindly overwrite whatever is on remote. --force-with-lease aborts if someone else has pushed since your last fetch — it is a safety check that turns a footgun into a reliable tool.

Putting it together with the earlier lessons

Lesson 1 showed you that rebase creates new commit hashes. Lesson 2 told you the deciding question: is the branch shared? This lesson adds the tools to act confidently:

  • Golden rule keeps teammates safe.
  • git reflog keeps your own work safe.
  • --abort lets you back out of a messy rebase at any moment.

The fear of rebase is usually fear of the unknown. After one or two real rebases — with this checklist in hand — it becomes routine.

Check your understanding

Answer from memory before clicking.

Rebase paused with a conflict. After manually editing the file to resolve it, what is the correct next sequence?

  • git commit -m "resolve" then git rebase --continue
  • git add <file> then git rebase --continue
  • git rebase --continue then git add <file>
  • git merge --continue then git rebase --continue

During a rebase, git rebase --continue takes the staged files and uses them to finish creating the current commit. You must stage with git add first so git knows what the resolved content is. Running git commit manually would interfere with the rebase's own commit management — do not do it.

You rebased your feature branch and immediately realised the result is wrong. You want to return to the state before the rebase. Which command exposes the pre-rebase commit SHA?

  • git log --all shows all unreachable commits
  • git status shows the previous HEAD position
  • git reflog lists every past HEAD position
  • git stash list stores a pre-rebase snapshot

git reflog records every move HEAD has made — commits, merges, rebases, checkouts, and resets. After a rebase, the entry labelled rebase (start) marks the exact commit you were on before rebase began. From there, git reset --hard HEAD@{n} restores you completely.

You pushed your feature branch three days ago. A teammate may have pulled it to review. Can you safely rebase it now?

  • Yes — rebase only rewrites your local copy
  • No — rebasing a shared branch breaks others' copies
  • Yes — git will automatically sync the new hashes
  • No — rebase simply cannot run on a pushed branch

The moment a teammate pulled your branch, it became shared. Rebasing rewrites the hashes; your remote now has different hashes from their local copy. They cannot simply git pull — their history has diverged. This is the golden rule in action: once pushed and potentially pulled, use merge.
