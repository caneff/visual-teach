Merge vs Rebase · Lesson 3 of 3

Rebase without fear

The shape and the decision are yours. The last thing between you and a confident rebase is knowing the one rule that keeps it safe — and that nothing you do is unrecoverable.

⏱ ~15 min Needs: L1 + L2 Intermediate Lesson 3 of 3

By the end of this lesson you'll be able to:

  • State the golden rule and know exactly why it exists.
  • Undo any rebase with git reflog — your work is never gone.
  • Resolve a mid-rebase conflict, or bail out cleanly with --abort.
  • Run a private-branch rebase end to end from a checklist.
Why this matters for your goal: this is the lesson that retires the fear for good. "I'll lose my work" stops being true the moment you've used the reflog once; "I'll clobber a teammate" stops being a risk the moment the golden rule is a reflex.

1The golden rule

There is exactly one rule that separates a safe rebase from a harmful one. It's worth memorizing word for word, straight from the Pro Git book:

The golden rule of rebasing
"Do not rebase commits that exist outside your repository and that people may have based work on." — Pro Git, §3.6

Rebasing makes new commits with new hashes (Lesson 1) and abandons the originals. If you're the only one who had those originals, no harm done. But if a teammate already pulled them, their history still points at the old hashes — and your rewritten branch no longer contains them. Their next pull turns into a tangle of duplicated commits and conflicts. That's the entire danger, and the one question from Lesson 2 ("has anyone else got these commits?") is how you stay on the right side of it.

Private rebase, public update. When you do rebase a branch you'd pushed (but nobody depends on), the remote still holds the old commits — so an ordinary push is rejected. Update it with the lease-guarded force, which refuses to run if someone else pushed in the meantime:
safe remote update after a private rebase 
git push --force-with-lease

2Your safety net: git reflog

Here's the fact that makes "I'll lose my work" simply false. Every time HEAD moves — every commit, checkout, merge, and rebase — git writes a line to the reflog. Your pre-rebase commits aren't deleted; they're just no longer pointed at by a branch. The reflog still remembers exactly where they were, for weeks. [git-reflog]

undo a rebase you regret 
git reflog                       # find where you were before the rebase
git reset --hard HEAD@{4}        # move the branch back to that exact commit
git reflog
7b1e44a HEAD@{0}: rebase (finish): returning to feature/login
d4a90c2 HEAD@{1}: rebase: Add login form
e5f8a90 HEAD@{2}: rebase (start): checkout main
3ab77d1 HEAD@{4}: commit: Add remember-me box   <- pre-rebase tip
9f0c4e2 HEAD@{5}: commit: Add login form
Read that line HEAD@{4}. It's your branch exactly as it stood before the rebase — old hashes and all. git reset --hard 3ab77d1 (or HEAD@{4}) puts you back as if the rebase never happened. There is no rebase you can't walk back this way.

3When a rebase hits a conflict

Rebase replays your commits one at a time. If a commit doesn't apply cleanly onto the new base, git pauses mid-rebase and hands you the conflict — it does not lose anything and it doesn't finish behind your back. You have three moves:

resolve and carry on 
# 1. git paused: CONFLICT in src/login.js
#    open the file, fix the <<<< ==== >>>> markers, save
git add src/login.js
git rebase --continue            # replay the next commit
the other two exits
git rebase --abort
# undo the whole rebase, return to the
# exact pre-rebase state. Always safe.

git rebase --skip
# drop the commit being replayed and
# move on. Use rarely, and on purpose.
Lost mid-rebase? --abort is the panic button. It returns your branch to exactly where it was before git rebase — no reflog needed. Reach for it any time a rebase gets confusing; you lose nothing but the in-progress attempt.

4The whole thing, as a checklist

This interleaves everything: the one question from Lesson 2, the straight-line shape from Lesson 1, and the safety net above. Tick it off the next time you rebase a private branch — your progress is saved in this browser.

  • Ask the one question Is this branch private — does anyone else have these commits? If yes, stop and merge instead. If no, rebase away.
  • Get the latest base Run git fetch origin so you rebase onto the real, current main.
  • Rebase On your branch, run git rebase origin/main. Your commits replay on top.
  • Handle any conflict Fix the markers, git add the file, git rebase --continue. Confused? git rebase --abort and you're exactly back where you started.
  • Verify the shape Run git log --graph --oneline. You should see one straight line — your commits sitting on top of main, no merge commit (Lesson 1).
  • Update the remote safely Only because it's private: git push --force-with-lease. Never plain --force.
  • Know your undo If anything looks wrong later: git reflog, find the pre-rebase line, git reset --hard HEAD@{n}. Nothing was ever lost.
