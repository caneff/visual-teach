Git merge vs. rebase · Lesson 3

Rebase without fear

The mechanics that turn rebase from scary to routine: one rule that keeps you safe, how to walk through a mid-rebase conflict, and the undo button that means you can never truly lose work.

⏱ ~15 min Needs: L1 + L2 Intermediate Lesson 3 of 3

By the end of this lesson you'll be able to:

  • State and apply the golden rule that keeps a rebase safe
  • Resolve a conflict mid-rebase with --continue / --abort
  • Recover a "lost" commit with git reflog — your safety net
  • Run a clean rebase end-to-end from a checklist
Why this matters for your goal: this is the lesson that retires the fear. Once you know the one rule and the undo button, rebase becomes a tool you reach for on purpose — not a trap you avoid.

1The golden rule (the only one you must keep)

From Lesson 2, distilled to a sentence you should be able to recite:

Never rebase commits that have been pushed and shared. Rebase builds new commits (Lesson 1) and abandons the old ones. If someone else holds the old ones, your history and theirs split apart — and merging the mess back together is exactly the pain you were trying to avoid.

Keep the rule and the worst case for a private branch is a few minutes of conflict resolution — fully reversible. Break it and you've handed your team a puzzle. Everything below assumes you've already confirmed the branch is yours to rewrite.

2When a rebase stops on a conflict

Replaying a commit can hit a line that also changed on the new base. Git pauses the rebase and hands the wheel to you. This is normal — not a failure. You have three moves:

git add → --continue --abort --skip

the mid-rebase loop 
# git stops and tells you which file conflicts:
git status                 # see what's unmerged

# open each file, find the conflict markers, and edit to the final result:
#   <<<<<<< HEAD            (what's on the new base)
#   =======
#   >>>>>>> your commit     (what your commit changed)

git add path/to/resolved-file   # mark this file settled
git rebase --continue           # replay the next commit

# went sideways? this returns you to EXACTLY where you started:
git rebase --abort

# this particular commit is now redundant? drop just it:
git rebase --skip
The rhythm is a loop: fix → git add → git rebase --continue, once per conflicting commit, until git says the rebase is done. You're resolving one replayed commit at a time, not the whole branch at once — which is why each step is small.
Escape hatch you can always trust: git rebase --abort. At any point mid-rebase — confused, conflicts piling up, lost your nerve — this puts the branch back exactly as it was before you started. Nothing is committed to the new shape until the rebase finishes, so there's always a clean way out.

3Your safety net: git reflog

Here's the fact that should end the fear of "losing work": git almost never deletes a commit immediately. Even after a rebase abandons the old commits, or a bad reset, they sit on disk for weeks. The reflog is git's private log of everywhere HEAD has been — every commit, rebase, reset, checkout — and it can take you back.

Find where you were 
git reflog
What it shows
f6a1b2c HEAD@{0}: rebase finished
9d4e5f6 HEAD@{1}: rebase: checkout main
a1b2c3d HEAD@{2}: commit: add validation
7c8d9e0 HEAD@{3}: commit: add form
...

See the commit you thought was gone (say a1b2c3d)? Bring it back. To move your current branch onto it:

recover a lost commit 
# inspect it first without committing to anything:
git checkout a1b2c3d

# or hard-reset your branch back to that exact state:
git reset --hard a1b2c3d

# safest: park it on a new branch so nothing else moves:
git branch rescue a1b2c3d
The takeaway: a rebase you regret is not a loss — it's a git reflog away from undone. This is the safety net that lets you rebase boldly on your private branches.

4If you already pushed your (private) branch

Rebasing a branch you'd pushed means your local history no longer matches the remote, so a normal push is rejected. Update it with --force-with-lease — never a bare --force:

push a rebased private branch 
git push --force-with-lease
--force-with-lease refuses the push if the remote has commits you haven't seen — i.e. if someone else pushed in the meantime. That check is your last guard on the golden rule: if it blocks you, the branch wasn't as private as you thought. Stop and reassess.

5The safe-rebase checklist

Put it all together. This is the workflow to follow every time — your progress is saved, so you can use it live on a real branch right now.

  • Ask the golden-rule question Are these commits pushed and shared with anyone? If yes, stop — merge instead. Only continue for private work.
  • Start from a clean tree Run git status. Commit or git stash any loose changes so the rebase starts from a known state.
  • Get the latest base and rebase Fetch, then from your branch run git rebase main (or git fetch && git rebase origin/main).
  • Resolve any conflict, one commit at a time Edit the conflicted files, then git add them and git rebase --continue. Repeat until git reports the rebase is finished.
  • Lost or confused? Abort At any point, git rebase --abort returns the branch to exactly where it began. No harm done.
  • Verify the new shape Run git log --oneline --graph (from Lesson 1) and confirm you see one clean line.
  • Update the remote safely (only if pushed) If this private branch was already pushed, git push --force-with-lease — never a bare --force.

6Interleaved recall — the whole course at once

These deliberately mix all three lessons. Mixing is harder than reviewing one topic — and that difficulty is exactly what builds durable memory. Answer before revealing.

(L1) After a rebase, why can't a teammate's old copy of your branch fast-forward to your new one?
Your replayed commits have new hashes — they're different commits from the ones your teammate holds. The two histories no longer share a tip to fast-forward to; they've diverged.
(L2) Your branch is pushed but only you have ever touched it. Merge or rebase to get the latest main?
Rebase. "Shared" means others have based work on it — a branch only you touch is still private, so rebasing is safe and keeps history linear.
(L3) You're three conflicts into a rebase and thoroughly lost. What's the single safest command?
git rebase --abort — it restores the branch to exactly its pre-rebase state. If you'd already finished and then regretted it, git reflog gets you back instead.
