Rebase without fear

The fear was never really about rebase — it was about not knowing how to get back. This lesson gives you a safe workflow, the conflict dance, and the undo button. After this, rebase is just a tool.

⏱ ~15 min Needs: Lessons 1–2 Intermediate Lesson 3 of 3

By the end of this lesson you'll be able to:

  • Run a rebase through a safe, repeatable checklist.
  • Resolve a mid-rebase conflict with --continue / --abort.
  • Undo any rebase with git reflog — your safety net.
Why this matters for your goal: This is the lesson that retires the anxiety. You'll rebase a local branch to tidy it before a PR — the exact win in your mission — and you'll have proof that nothing can be lost. It interleaves everything: the shapes from Lesson 1 and the one question from Lesson 2, now in your hands.

1First, the gate (Lesson 2, one more time)

Every safe rebase starts by clearing the golden rule. If the answer here is "shared", stop and merge instead — no checklist will save a rebase of commits other people hold.[1]

Gate — Are these commits pushed and shared? If yes, merge — you're done here. If no, they're yours alone and rebase is safe; continue below.

2The safe rebase workflow

Work down this list the first several times — it's saved to your browser, so your ticks persist if you come back. The goal is a routine boring enough that rebase stops feeling risky.

  • Clear the gate Confirm the branch is private (not pushed, or pushed but only yours). Shared → merge and stop.
  • Get a clean working tree Commit or git stash any work in progress. Rebase wants nothing uncommitted in the way.
  • Fetch the latest base Run git fetch origin so you're replaying onto the real current main, not a stale copy.
  • Rebase On your branch, run git rebase origin/main. Git replays your commits onto the new tip (Lesson 1's straight line).
  • Handle conflicts if they appear Resolve, git add each file, then git rebase --continue. Unsure at any point? git rebase --abort returns you to the start, untouched. (See §3.)
  • Verify the shape Run git log --oneline --graph and confirm a clean line with your commits on top of main.
  • Push (force only your own branch) Never pushed it? A normal git push. Already pushed this private branch? You rewrote it, so use git push --force-with-lease — it refuses if someone else pushed in the meantime.
Why --force-with-lease and not --force? Plain --force overwrites the remote no matter what. --force-with-lease first checks the remote still points where you last saw it — so if a teammate did push, it stops instead of clobbering them. It's the seatbelt for the one push a rebase needs.[2]

3The conflict dance

A rebase replays commits one at a time, so a conflict pauses on the exact commit that collided. You get three moves, and only three. Here's a real run:

a rebase pauses on a conflict 
$ git rebase origin/main
Auto-merging config.yml
CONFLICT (content): Merge conflict in config.yml
error: could not apply 1f3c2ab... raise timeout
# git stopped on ONE commit. Fix it, or bail.

$ git status                 # which file? which commit?
$ # ...edit config.yml, keep the right lines, remove the markers...
$ git add config.yml         # mark this file resolved
$ git rebase --continue      # replay the next commit

# Not sure? This always works and is totally safe:
$ git rebase --abort         # undo everything, back to the start
The escape hatch is real. git rebase --abort resets HEAD to exactly where it was when the rebase started — as if you never ran it.[3] You can always back out of a rebase in progress. That alone removes most of the fear.

The three moves, and nothing else:

edit + git add + --continue --skip (drop this commit) --abort (undo it all)

4The undo button: git reflog

What if the rebase finished and only then you realize it went wrong? You're still safe. Git keeps a private log of every place HEAD has been — the reflog — and you can jump back to any of them.[4]

rewind to before the rebase
$ git reflog
a1b2c3d HEAD@{0}: rebase finished: returning to refs/heads/feature
9f8e7d6 HEAD@{1}: rebase: raise timeout
1122334 HEAD@{2}: checkout: moving to feature   # <- the good state, before rebase
...

$ git reset --hard HEAD@{2}    # HEAD jumps back; your old commits return

The reflog is a time machine for HEAD. Even a "finished" bad rebase is one reset --hard HEAD@{n} away from undone. Entries stick around for ~90 days, so there's no rush.

6Earn the win: tidy a real branch

Do this once on a private branch and the fear is gone for good. Safe, reversible, and exactly the mission task:

tidy a local feature branch before a PR 
git switch feature           # a branch only you have
git status                   # clean tree? (commit or stash first)
git fetch origin             # get the real current main
git rebase origin/main       # replay your commits on top — clean line
# conflicts? resolve, git add, git rebase --continue (or --abort)
git log --oneline --graph    # admire the straight line
# if this branch was never pushed:
git push -u origin feature
The whole course in one breath: two shapes (L1), one question (L2), and a safety net (this lesson). Merge keeps the fork; rebase makes a clean line of new commits; never rebase shared work; and reflog means you can always get back.
