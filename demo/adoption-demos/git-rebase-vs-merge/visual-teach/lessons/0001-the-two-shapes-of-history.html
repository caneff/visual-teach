Git merge vs. rebase · Lesson 1

The two shapes of history

Before you can choose between merge and rebase, you need to see what each one does to your commit history. By the end of this page you'll be able to draw both shapes from memory.

⏱ ~10 min Needs: basic commit / merge Beginner Lesson 1 of 3

By the end of this lesson you'll be able to:

  • Explain what a commit really is: a snapshot that points at its parent
  • Draw the branching graph that merge produces
  • Draw the linear graph that rebase produces — and say why the commit hashes change
Why this matters for your goal: the fear of rebase ("it rewrites history, I'll lose work") dissolves once you can picture exactly what it moves. This lesson builds that picture. No decisions yet — just the mental model everything else rests on.

1One idea: a commit points at its parent

A commit is a snapshot of your whole project plus a pointer to the commit that came before it — its parent. Follow the parent pointers backward and you've walked your entire history. So history isn't a list; it's a graph of commits linked child-to-parent.

A branch (like main or feature) is just a sticky note naming the latest commit on a line of work. When you commit, the note slides forward to the new commit. Everything merge and rebase do is rearrange this graph and move those notes — nothing more mysterious than that.

The key fact for later: a commit's identity — its hash, like a1b2c3d — is computed from its contents and its parent. Change the parent and you necessarily get a different commit, with a different hash. Hold onto this; it's the whole reason rebase "rewrites history."

2The starting point: a branch that diverged

Here's the situation you hit every day. You branched feature off main, did some work (e5, f6) — and meanwhile a teammate moved main forward (c3, d4). The two lines have diverged:

  • main commits
  • feature commits

The shared ancestor is b2. From here you must somehow combine the two lines. merge and rebase are the two ways to do it.

3Merge: keep both lines, tie them together

git merge feature (run while on main) creates one new merge commit that has two parents — the tip of each line. Nothing already in history moves or changes. The real, branching shape of what happened is preserved exactly:

  • main commits
  • feature commits
  • the new merge commit (two parents)

m7 is brand-new; a1…f6 are untouched. The graph keeps its fork — you can always see that feature was developed in parallel.

Fast-forward aside. If main had not moved on (no c3, d4), merging would just slide the main note forward onto f6 — a "fast-forward," no merge commit at all. A merge commit only appears when the two lines genuinely diverged, as above.

4Rebase: replay your work onto the new base

git rebase main (run while on feature) does something different: it takes your commits (e5, f6), sets them aside, fast-forwards your branch to the tip of main (d4), and replays your changes one by one on top. Because each replayed commit now has a new parent, each gets a new hash — e5 becomes e5′, f6 becomes f6′. The result is one straight line:

  • main commits
  • replayed feature commits (new hashes)
  • the originals, now abandoned

The history reads as one clean line, as if you'd started your work from d4 all along. The fork is gone. e5′ and f6′ contain the same changes as e5 and f6 — but they are different commits.

"Rewrites history" sounds scary, but now you can see exactly what it means: rebase doesn't edit your old commits — it builds new ones with the same changes and a new parent, then points your branch at them. The faded e5/f6 still exist on disk for a while (you'll meet git reflog, your safety net, in Lesson 3).

5See it in your own terminal

One command shows you these shapes for any repository. Run it after a merge and after a rebase and watch the graph change:

the one command worth memorizing 
git log --oneline --graph --all
After a merge
*   m7 Merge branch 'feature'
|\
| * f6 add validation
| * e5 add form
* | d4 fix header
* | c3 tweak nav
|/
* b2 base
* a1 init
After a rebase
* f6′ add validation
* e5′ add form
* d4 fix header
* c3 tweak nav
* b2 base
* a1 init

The merge graph forks and rejoins (|\ … |/); the rebase graph is one straight column. That shape is the difference.

6Recall — say it before you reveal it

Don't peek. Produce each answer out loud or on paper first — the effort of retrieval is what makes it stick.

What does a merge commit have that an ordinary commit doesn't?
Two parents — the tip of each branch being joined. That second parent is what preserves the fork in the graph.
Why do rebased commits get brand-new hashes?
A commit's hash is computed from its contents plus its parent. Rebase replays each commit onto a new parent, so each becomes a new commit with a new hash — same changes, new identity.
In one word each: what shape does merge produce, and what shape does rebase produce?
Merge → branching (forks preserved). Rebase → linear (one straight line).

✦ What you earned

You can now draw both shapes: merge adds a two-parent commit and keeps the fork; rebase replays your commits onto a new base, producing a straight line with new hashes. That's the entire mechanical difference — and the foundation for choosing between them.
