Merge vs Rebase — cheat sheet & glossary

The compressed essence of the course on one page. Built to be printed and pinned. Lessons: 1 · 2 · 3.

The decision, in one question

Has anyone else got these commits?
No → private branch → git rebase (tidy line, safe to rewrite).
Yes → shared branch → git merge (never rewrite shared history).
Note: pushed ≠ shared. A solo PR branch nobody builds on is still private — rebase it, then push --force-with-lease.

The two shapes

merge Keeps the fork; adds a merge commit with two parents. Originals untouched (same hashes). History shows the branches met.
rebase Replays your commits onto a new base → one straight line. New commits, new hashes; originals abandoned (kept in the reflog). Looks like it was always linear.

Command quick reference

merge — integrate, preserve the fork 
git checkout main
git merge feature/login          # create a merge commit on main
git merge --no-ff feature/login  # force a merge commit even if fast-forward
git merge --abort                # cancel a merge that hit conflicts
rebase — replay onto a new base, straighten history 
git fetch origin
git checkout feature/login
git rebase origin/main           # replay your commits on top of latest main
git pull --rebase                # fetch + rebase in one step
git push --force-with-lease      # update a PRIVATE pushed branch after rebase
mid-rebase — three exits 
# conflict? fix the file, then:
git add <file>
git rebase --continue            # replay the next commit
git rebase --abort               # undo everything, back to pre-rebase state
git rebase --skip                # drop the current commit (rare, deliberate)
recovery — nothing is ever lost 
git reflog                       # list every place HEAD has been
git reset --hard HEAD@{4}        # move branch back to a pre-rebase point
git reset --hard 3ab77d1         # ...or to a specific old commit hash

The golden rule

Do not rebase commits that exist outside your repository and that people may have based work on. — Pro Git §3.6  Break it and teammates inherit duplicated commits and conflicts on their next pull.

Glossary

merge commit A commit with two parents, created by git merge to tie two branches together. The visible record that they met.
fast-forward When the target branch hasn't diverged, merge just slides the branch pointer forward — no merge commit. --no-ff forces one anyway.
three-way merge The merge git performs when branches have diverged, using both tips plus their common ancestor. Produces a merge commit.
rebase Re-create your commits on top of a different base commit. "Re-base" = give them a new base.
replay How rebase works: it applies each of your commits, one at a time, onto the new base — making a new commit (new hash) each time.
base / upstream The commit your branch is built on top of. git rebase main changes the base to the tip of main.
diverged Two branches that each have commits the other lacks — they share an ancestor but have forked since. This is when the merge-vs-rebase choice appears.
commit hash The SHA-1 id of a commit. Rebase produces new hashes because each replayed commit is a new object.
HEAD A pointer to where you are right now (usually the tip of the current branch). HEAD@{n} in the reflog means "where HEAD was n moves ago".
reflog A local log of every position HEAD has held. Your undo history and safety net — abandoned commits live here for weeks.
golden rule Never rebase commits others may have based work on. The single rule that keeps rebase safe.
--force-with-lease A force-push that first checks nobody else pushed since you last fetched. Use it (never plain --force) to update a private branch after rebasing.
private vs shared Private = only you have these commits. Shared = others have pulled them or build on them. The distinction the golden rule turns on — not whether you pushed.
