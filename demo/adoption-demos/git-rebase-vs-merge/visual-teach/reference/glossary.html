Git merge vs. rebase · Reference

Git history glossary

The canonical vocabulary for this course. Every lesson uses these terms in exactly this sense. Print it; keep it by the keyboard.

Read the first group once and the rest will click — almost every term here is just a consequence of one fact: a commit's identity includes its parent.

Core graph

Commit
A snapshot of the whole project plus a pointer to its parent and a hash that identifies it. The atomic unit of history.
Parent
The commit that came immediately before a given commit. Following parents backward walks the entire history. Avoid: "previous version"
Hash
A commit's identifier (e.g. a1b2c3d), computed from its contents and its parent. Change either and the hash changes — it's a different commit. Avoid: "commit number", "id"
Branch
A movable name pointing at the latest commit on a line of work (e.g. main, feature). Committing slides the name forward. Avoid: "copy of the code"
HEAD
A pointer to the commit (usually via a branch) you currently have checked out — "where you are now."
Diverged
The state where two branches have each gained commits since their shared ancestor, so neither is simply ahead of the other.

Merge

git merge
Combines two branches by creating one new merge commit that joins them. Existing commits are never changed; the branching shape is preserved.
Merge commit
A commit with two parents — the tip of each branch being merged. Its second parent is what records the fork in the graph.
Fast-forward
When the target branch hasn't diverged, merging just slides its name forward to the other tip — no merge commit is created. Avoid: "auto-merge"
Three-way merge
The merge git performs when branches have diverged: it compares both tips against their shared ancestor to build the result.

Rebase

git rebase
Moves a line of commits to a new base by replaying them on top of it, producing a linear history.
Replay
Re-applying each commit's changes onto a new parent, one at a time. Because the parent changes, each replayed commit gets a new hash — same changes, new identity. Avoid: "moving commits", "re-running"
Rewriting history
What rebase does: it doesn't edit old commits, it builds new ones and points the branch at them, abandoning the originals. Avoid: "destroying history", "deleting commits"
Linear history
A history with no merge commits — one straight line of commits. The upside rebase optimises for.
The Golden Rule of Rebasing
Never rebase commits that have been pushed and shared. Rewriting shared commits splits your history from everyone else's.
Shared
A commit is "shared" once someone else may have based work on it — not merely once it's pushed. A branch only you have ever touched is still private. This distinction decides merge vs. rebase.

Conflicts & recovery

Conflict
When a replayed (or merged) change touches a line that also changed on the base, git pauses and asks you to choose the final result.
--continue / --abort / --skip
Mid-rebase controls: resume after resolving (--continue), cancel and restore the pre-rebase state (--abort), or drop the current commit (--skip).
git reflog
Git's private log of every position HEAD has held. Lets you find and recover a commit after a rebase, reset, or bad checkout — the safety net. Avoid: "git history", "the log"
--force-with-lease
A safe force-push: it updates a rebased branch on the remote but refuses if someone else has pushed in the meantime. Always prefer it over a bare --force.
