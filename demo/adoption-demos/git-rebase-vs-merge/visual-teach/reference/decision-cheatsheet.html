Git merge vs. rebase · Reference

Decision & recovery cheat sheet

One page. The question that decides every case, the everyday defaults, and the exact commands for rebasing and recovering. Print it.

The one question

Have these commits been pushed and shared with anyone else?
YES → merge Rewriting shared commits splits your history from theirs. Bring changes in with git merge; never rebase the shared branch.
NO → rebase ok The commits are private. Rebase freely for a clean, linear history — git rebase --abort and git reflog have your back.
"Shared" means someone may have based work on it — not merely that it's pushed. A branch only you have ever touched is private.

Everyday defaults

SituationChoiceWhy
Tidy a local branch before a PR rebase Private — free to rewrite; reviewers get a clean line.
Update your own branch with latest main rebase Lands your work on the newest base, linear history.
Integrate main into a shared branch merge Others hold those commits — rebasing would break them.
Merge a finished feature into main merge main is canonical and shared; never rebase it.
Your team mandates a house style follow it A team convention overrides every default above.

Rebase, end to end

safe rebase of a private branch 
git status                       # 1. clean tree? stash/commit loose work
git fetch                        # 2. get the latest base
git switch feature               #    (be on the branch you're rebasing)
git rebase origin/main           # 3. replay your commits onto main

# 4. on a conflict, per commit:
git status                       #    which files conflict?
#    edit files, removing <<<<<<< ======= >>>>>>> markers
git add path/to/file
git rebase --continue            #    replay the next commit

git rebase --abort               # bail out — restores pre-rebase state
git log --oneline --graph        # 5. verify: one clean line
git push --force-with-lease      # 6. only if it was already pushed

Recover anything

the undo button 
git reflog                       # every place HEAD has been
git branch rescue <hash>         # safest: park the lost commit on a branch
git reset --hard <hash>          # or move current branch back to it
git checkout <hash>              # or just look at it first
Never rebase a shared branch, and never push --force (bare) over one. Use --force-with-lease, which refuses if someone else has pushed.
