merge vs rebase — Quick Reference

The compressed essence of the course: glossary, the decision rule, the commands, and how to bail out. Built to print on one or two pages and live next to your keyboard.

Companion to Lessons 0001, 0002, and 0003 · verified 2026-06-25

1The two shapes, at a glance

main commit feature commit merge commit (2 parents) replayed copy (new hash)

merge ties two lines together with a merge commit and keeps the real, branching history. rebase copies your commits onto a new base, one after another, producing a straight line of brand-new commits.

2Glossary

commit A snapshot plus a pointer to its parent(s), named by a unique hash. Commits are immutable — you never "edit" one, you replace it.
branch A movable label pointing at one commit (its tip). Cheap; it is just a name.
HEAD Where you are right now — the commit your working tree is built on.
merge commit A commit with two parents that joins two branches. The fingerprint that a merge happened.
fast-forward A "merge" with nothing to reconcile: the branch label just slides forward. No merge commit is created.
replay What rebase does: re-apply each of your commits onto a new base, creating a new commit (new hash) for each.
base The commit your branch was built on. Rebase changes it — that is the whole point.
linear history One straight line of commits, no forks. The shape rebase produces.
reflog A local log of everywhere HEAD has been. Your undo button — even after a rebase.
force-push git push --force: overwrite the remote branch with your rewritten history. Safe on a private branch; dangerous on a shared one.

3The decision rule

One question decides it: have these commits been pushed and possibly used by someone else? Yes → merge. No → rebase is safe (and usually tidier).
Situation Use Why
Tidy up my local feature branch before a PR rebase Commits are private; a clean line is easier to review.
Update my branch with the latest main rebase Replays your work on top of new main; no noise merge commit.
Integrate main into a long-running shared branch merge Others have the branch — rewriting it would clobber their work.
Land a finished feature into main merge Records honestly that a feature was integrated, when.
Anything already pushed and shared merge The golden rule: never rebase shared commits.

4Command reference

everyday commands
# Merge another branch into the one you're on
git merge main              # may create a merge commit
git merge --no-ff feature   # force a merge commit even if fast-forward possible

# Rebase the current branch onto a new base
git rebase main             # replay my commits on top of main
git pull --rebase           # update: fetch, then rebase instead of merge

# Mid-rebase, after a conflict
git status                  # see what's conflicted
git add <file>              # mark each conflict resolved
git rebase --continue       # resume replaying
git rebase --abort          # bail out — back to exactly where you started

# The safety net
git reflog                  # list everywhere HEAD has been
git reset --hard HEAD@{1}   # jump HEAD back to a previous position

5The golden rule & recovery

Golden rule — Do not rebase commits that exist outside your repository and that people may have based work on. (Pro Git, "The Perils of Rebasing.") Translation: never rebase anything you've already pushed and shared.
Recovery in three lines — A rebase went wrong? You are not stuck. If you're mid-rebase: git rebase --abort. If it already finished and looks wrong: git reflog, find the line just before the rebase, then git reset --hard HEAD@{n}. Reflog keeps entries for ~90 days.
