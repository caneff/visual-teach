Merge vs. Rebase — Cheat Sheet

The compressed essence of the course. Built for quick reference and for printing. Lessons: 1 · 2 · 3.

The one-line difference

Merge joins two histories with a new merge commit and keeps the real, branching graph.
Rebase replays your commits one by one onto a new base, producing a straight line and new commit hashes.

Decision table

SituationUseWhy
Tidy a local feature branch before opening a PR rebase Commits are private; linearising makes the PR easy to read.
Update your feature branch with the latest main rebase (if branch unshared)
else merge		 Rebase keeps a clean line. If others share the branch, merge instead.
Integrate a finished feature into a shared / long-running branch merge Preserves true history; doesn't rewrite commits others have.
Bring a feature into main and you want the merge recorded merge --no-ff Keeps an explicit, auditable merge commit.
Anyone else has already pulled these exact commits merge (never rebase) The golden rule — see below.
The Golden Rule Never rebase commits that exist outside your repository and that others may have based work on. Rewrite only private, un-pushed history. — Pro Git, "The Perils of Rebasing"

Conflict-during-rebase loop

git rebase main
# CONFLICT in app.js
#   1. open app.js, resolve the <<< === >>> markers
#   2. git add app.js
#   3. git rebase --continue        # next commit replays
# repeat until done — or at any point:
git rebase --abort                  # nothing lost, original branch restored
Safety net A rebase only ever moves a branch pointer. The original commits stay in the reflog for ~90 days. git reflog + git reset --hard gets them back. You cannot truly lose committed work to a rebase.

Glossary

Commit
A snapshot of the whole project plus a pointer to its parent(s), identified by a hash. Immutable: change anything and you get a new hash.
Branch
A movable pointer to one commit (its tip). Cheap — just a name.
HEAD
A pointer to the commit (usually via a branch) you currently have checked out.
Merge commit
A commit with two parents, tying two histories together. The visible signature of a merge.
Fast-forward
When the target hasn't diverged, merge just slides the branch pointer forward — no merge commit is created.
Rebase
Re-applying a sequence of commits onto a new base commit. Produces new commits (new hashes) with the same changes.
Replay
The act, during a rebase, of re-creating each of your commits one at a time on top of the new base.
Linear history
A history with no merge commits — a single straight line. The typical result of rebasing.
Reflog
A local log of every position HEAD and your branch tips have held. The undo button for "I rewrote history and panicked."
