The Golden Rule

Never rebase commits that have been pushed to a shared remote branch.

Rebase rewrites commit hashes. If others have built on your original hashes, rewriting them leaves their work pointing to commits that no longer exist in your branch. See Lesson 1 for why hashes change, and Lesson 3 for reflog recovery.

Decision Guide

Situation Use Reason
Local feature branch, not yet pushed git rebase main Private — safe to rewrite; gives a clean linear PR
Shared branch (develop, staging, release) git merge main Shared — never rebase; merge is always safe
Already-pushed PR branch, need latest main git merge main (safe default) Once pushed, merge avoids disrupting the open PR
Integrate completed feature into main Either (team convention) Merge preserves context; rebase + FF gives linear log
Hotfix into both main and release git merge into both Both branches are shared; merge is safe

The Two History Shapes

Merge Rebase
Creates merge commit? Yes — with two parents No
Existing commit hashes change? No — only M is new Yes — every replayed commit gets a new hash
History shape Branching graph (diamond) Linear
Rewrites history? No Yes
Safe on shared branches? Always Never 
Starting point:
  A -- B -- C        (main)
             \
              D -- E (feature)

After git merge feature:          After git rebase main (from feature):
  A -- B -- C -- M  (main)          A -- B -- C        (main)
             \   /                            \
              D-E   (feature)                 D'-- E'   (feature, new hashes)

Commands

git merge <branch>
Creates a merge commit joining the current branch and <branch>. Safe on shared branches; never rewrites existing commits.
git merge --no-ff <branch>
Forces a merge commit even when a fast-forward is possible. Preserves the branching record in the graph.
git rebase main
Replays the current branch's commits on top of main. Use only on local/private branches.
git rebase --continue
After staging resolved conflict files, proceeds to the next commit in the rebase.
git rebase --abort
Cancels the in-progress rebase and returns the branch to its pre-rebase state.
git reflog
Shows every position HEAD has pointed to, including pre-rebase branch tips. Primary recovery tool.
git reset --hard HEAD@{N}
Restores the branch to where HEAD was N steps ago, as shown by git reflog.
git push --force-with-lease
Force-push after rebase; refuses if a teammate pushed new commits since your last fetch.
git log --graph --oneline
Shows commit history as ASCII graph. Diamonds = merge commits; straight line = rebase + fast-forward.

Conflict Workflow (Rebase)

  1. Git pauses and prints a CONFLICT message.
  2. Run git status — conflicted files appear under both modified.
  3. Open each conflicted file; resolve the <<<<<<< / ======= / >>>>>>> markers by editing to the desired state.
  4. git add <file> for each resolved file. Do not run git commit.
  5. git rebase --continue — repeat from step 2 if more conflicts appear.
  6. If stuck at any point: git rebase --abort returns everything to the pre-rebase state.

Reflog Recovery

# 1. See where HEAD has been
git reflog

# 2. Find the entry before your rebase started, e.g.:
#    3f2e1d0 HEAD@{3}: commit: Add login form    ← branch tip before rebase
#
# 3. Restore to that point
git reset --hard HEAD@{3}

# Reflog retains entries for 90 days by default.
# It is local — recover on the same machine where the rebase ran.

Glossary

Commit graph (DAG) The directed acyclic graph of all commits in a repository. Each node is a commit; each edge points to a parent commit.
Merge commit A commit with two parents. Created by git merge when joining two branches. Preserves the branching structure in the permanent history.
Fast-forward merge When the current branch is a direct ancestor of the target, git advances the branch pointer without creating a merge commit. Only possible when branches have not diverged.
Rebase Replays a sequence of commits onto a new base. Produces new commit objects with new hashes; results in a linear history with no branch-and-merge shape.
SHA-1 hash The 40-character identifier for a git object. Derived from its content, author, timestamp, and parent hash — so changing a commit's parent always produces a new hash.
Reflog A local log of every position HEAD has pointed to. The primary recovery tool after a rebase gone wrong. Retained for 90 days by default; local only, not pushed to remotes.
--force-with-lease A safer alternative to --force for push after a rebase. Rejects the push if the remote has commits you haven't fetched — protecting teammates' work from being overwritten.
Golden rule The one hard rule: do not rebase commits already on a shared remote branch. Everything else is recoverable; violating this creates collaborative chaos for everyone on the branch.
