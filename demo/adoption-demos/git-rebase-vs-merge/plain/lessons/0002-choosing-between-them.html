git merge vs rebase · Lesson 2 of 3

Choosing between them

You can already draw the two shapes (Lesson 1). Now the real skill: standing at the keyboard in a concrete situation and picking the right one — fast, and for a reason you can say out loud.

Your win for today: a one-question decision rule, drilled against real scenarios until the answer feels automatic.

The whole decision in one question

The deciding question

"Have these commits been shared with anyone else yet?"

If no (they're private to your local branch) → you're free to rebase for a clean, linear history.
If yes (pushed and others may have built on them) → merge, so you don't rewrite commits people already have.

That single question — private vs. shared — settles most cases. Everything below is just that rule applied to the situations you hit every week. [Atlassian]

A second axis: do you want the fork on the record?

When both options are safe, a taste question breaks the tie:

Lean rebase when…

  • The work is yours and unshared
  • You want history readable and linear
  • The branch is short-lived (a PR)
  • The intermediate fork isn't worth recording

Lean merge when…

  • The commits are shared / pushed
  • You want a true record of when lines joined
  • The branch is long-running and collaborative
  • The merge point itself is meaningful

The three scenarios you'll actually meet

① Tidy your local feature branch before opening a PR

Your commits live only on your machine. Meanwhile main drifted forward and your branch has gone stale. You want a clean line for review.

$ git switch feature
$ git rebase main      # replay your work on the latest main, linear & clean

→ Rebase. Private commits, short-lived branch, readability matters for the reviewer. Textbook rebase.

② Update your branch with the latest main mid-flight

Same situation, recurring: main moves while you work. To pull its changes in without a merge commit every time:

$ git pull --rebase origin main    # fetch, then replay your commits on top

→ Rebase (as long as your branch's commits aren't shared). Keeps your branch a clean line that's easy to eventually fast-forward.

③ Integrate main into a long-running shared branch

A release branch several teammates have pulled and committed to. Their commits are shared. Rebasing would rewrite history out from under them.

$ git switch release
$ git merge main       # add a merge commit; nobody's commits are rewritten

→ Merge. Shared commits, long-running branch, the integration point is real history worth keeping.

Decision drill — the feedback loop

This is the heart of the lesson. For each scenario, decide before clicking. Getting it wrong (then seeing why) builds the skill faster than getting it handed to you. Same idea, slightly different clothes each time — that's deliberate.

Pick merge or rebase.

Pick the command.

Say it back, in your own words

Without looking up: what's the one-question rule, and what does each answer tell you to do?

"Have these commits been shared with anyone yet?" No → rebase (private work, clean it up freely). Yes → merge (don't rewrite history others already have). When both are safe, pick rebase for a linear branch you'll discard, merge to record a meaningful join.
