Git merge vs. rebase · Lesson 2 of 3

Choosing between them

In Lesson 1 you learned the two shapes. Now the actual skill: given a real situation, pick merge or rebase and justify it in one sentence — fast, and without anxiety.

This lesson is mostly practice. Knowledge first (one small rule), then a decision feedback loop you work through.

The one question that decides it

Ask yourself "Has anyone else already pulled these exact commits?"
· No (the commits are private/local) → you're free to rebase for a clean, linear history.
· Yes (they're shared) → merge; rebasing would rewrite commits other people already have.

That single question resolves most cases. The guiding principle, from the Pro Git book: "Rebase local changes before pushing to clean up your work, but never rebase anything that you've pushed somewhere."1 A secondary preference breaks ties: rebase when you value a clean, linear, readable history; merge when you value an exact record of what happened (and an auditable merge commit).2

The three canonical situations

Situation Pick
Tidy a local feature branch before opening a PR rebase
Update your (unshared) branch with the latest main rebase
Integrate into a shared, long-running branch merge

Decision feedback loop

For each scenario, decide before you click. Immediate feedback — that's the loop that builds the skill.2 Answers are kept the same length so formatting gives nothing away.

Scenario 1

You've been hacking on a local branch nobody else has pulled. Before you open the PR you want a clean, readable history on top of the current main.

Yes. Private commits + want-it-clean → git rebase main. Linear history makes the PR trivial to review, and no one is affected.
The commits are private and you want it clean — that's exactly the rebase case.

Scenario 2

main moved ahead while you worked. Your branch is still local-only, and you just want the newest main underneath your commits.

Correct. git rebase main (or git pull --rebase) lifts your commits onto the latest main with no merge-commit noise — fine because the branch is unshared.
Merging works, but it litters an unshared branch with merge commits. Rebase is the cleaner choice while the branch is still private.

Scenario 3

Two teammates are committing to a long-running release branch every day. You need to bring main's latest fixes into it.

Right. Others have these commits, so rebasing would rewrite history out from under them. git merge main integrates safely and records the real picture.
A clean line is tempting, but the branch is shared — rewriting it breaks your teammates' clones. This is a merge.

Scenario 4 — the trap

You already pushed your feature branch and a reviewer pulled it. You now wish the history were a little cleaner.

Exactly. Once commits are pushed and pulled, they're public. Rewriting and force-pushing would clobber the reviewer's copy — the classic way to "lose work." Leave them, or merge. (More in Lesson 3.)
This is the trap. Those commits are shared — rebasing + force-pushing is how a teammate's work gets clobbered. The golden rule says don't.
One-sentence justifications — say each aloud, then check.
  • Local cleanup: "Rebase — the commits are private, so I can linearise freely."
  • Update unshared branch: "Rebase — replay my work onto the latest main with no merge noise."
  • Shared branch: "Merge — others have these commits, so I must not rewrite them."
  • Already pushed & pulled: "Merge — they're public now; rewriting would clobber a teammate."
