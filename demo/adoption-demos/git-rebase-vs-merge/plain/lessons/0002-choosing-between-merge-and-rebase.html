Lesson 2 of 3

Choosing Between Merge and Rebase

You know what each command does to the graph. This lesson builds the decision skill: given a real scenario, which one do you reach for?

The question that settles most cases

Have the commits on this branch been pushed somewhere that a teammate could have based work on?

Yes → Use merge. Rebasing rewrites your commits into new ones with new hashes. If your teammate already has the originals, their local history and yours will have diverged — two different commits claiming the same ancestry. Untangling that is painful. (Atlassian: The Golden Rule of Rebasing)

No → Rebase is available. A branch that only exists on your machine, or that only you push to and no one else has checked out, can be freely rebased. You are the only one who has ever seen those commit hashes.

One question eliminates most of the danger. The rest of the decision is about preference, legibility of history, and team conventions.

Five concrete scenarios

Scenario A — Tidy up a local branch before opening a PR

You have been working on feature/login for two days. The branch has never been pushed. You want to clean it up before code review.

Rebase. git rebase main brings your branch current with main and produces a clean linear diff that is easy for reviewers to read. Since the branch is local-only, no one else's history is affected.

Scenario B — Integrate a finished feature into main

A feature branch is approved and ready to land. You want a permanent record that "this feature shipped here."

Merge with --no-ff. git merge --no-ff feature/login forces a merge commit even when a fast-forward is possible, preserving the feature boundary in history. That boundary is useful for archaeology ("when did auth land?") and rollbacks. (git-merge docs: --no-ff)

Scenario C — Update a long-running shared branch

Three teammates push to develop daily. You need to integrate last week's hotfixes from main.

Merge. The branch is shared. git merge main adds a merge commit recording the integration event. It adds a node to the graph, but it doesn't rewrite anything anyone else has already pulled.

Scenario D — Your solo feature branch needs the latest main

You pushed your feature branch to GitHub yesterday for safekeeping, but only you commit to it and no one else has checked it out. You want to incorporate today's main changes.

Rebase — with one extra step. git rebase main, then git push --force-with-lease. --force-with-lease is safer than --force: it aborts if the remote has new commits you haven't fetched, which protects you from overwriting someone else's accidental push to your branch. (git-push docs: --force-with-lease)

Scenario E — Squash messy WIP commits before review

You pushed five "WIP", "fix", "oops" commits to your personal branch. You want to collapse them into one clean commit before the PR is reviewed.

Interactive rebase. git rebase -i HEAD~5 opens an editor where you mark each commit as pick, squash, or reword. Squash the WIP commits into the one real commit. Then force-push. The branch is yours and unmerged, so rewriting is safe.

The short-form rules

Situation Command Why
Branch is local-only, update from main before PR git rebase main Linear history, clean diff for reviewers
Shared branch needs latest main git merge main Never rewrite commits others have based work on
Feature lands in main, preserve boundary git merge --no-ff Merge commit captures "feature shipped here"
Squash WIP commits before review (private branch) git rebase -i HEAD~N Rewrite is safe; produce one clean commit

Decision practice

Each scenario below has one best answer. Work through each one before reading the explanation — the effort of deciding is what builds the reflex.

Your feature/login branch has never been pushed. You want to bring it current with main before opening a PR.

Rebase is the right call. The branch is local — no one else has based work on your commits. After rebasing, the PR diff will be clean and current with main, making review easier.

Three teammates are actively committing to develop. You need to bring in a hotfix from main.

Merge. Three people are committing to develop and have pulled its commits. Rebasing would rewrite the history they've based work on, creating divergence that's difficult to untangle. Cherry-pick could work in some situations but changes the commit hash of the hotfix.

You pushed five "WIP" commits to your personal feature branch. No one else works on it. You want to squash them into one tidy commit before code review.

Interactive rebase is the right tool. git rebase -i HEAD~5 opens an editor where you mark the WIP commits as squash — they collapse into the preceding commit. Force-push is safe because only you use this branch. (git merge --squash is used from the target branch to bring in a squashed copy; it doesn't reshape your branch.)

A feature branch is approved and ready to land in main. Your team wants a clear record in git log of when each feature was integrated.

--no-ff (no fast-forward) forces a merge commit even when one isn't strictly required. That commit says "this feature landed here, at this time, from this branch" — which is exactly the record your team wants. Fast-forwarding after a rebase would produce a straight line and lose that boundary.

You pushed your feature branch last week for safekeeping. Only you push to it, and your teammates have never mentioned checking it out. Today you want to update it from main.

Rebase is fine here — if you are confident no one else has checked out the branch. The key safeguard is --force-with-lease instead of --force: it refuses to push if the remote has new commits you haven't fetched, protecting you from accidentally overwriting someone else's accidental push. If you're ever unsure whether a colleague has the branch, default to merge.
