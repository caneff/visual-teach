git merge vs rebase · Lesson 3 of 3

Rebase without fear

You know the two shapes and you know when to reach for each. The last thing between you and a calm rebase is the fear: "what if I lose work, or clobber a teammate?" This lesson removes it — one rule to stay safe, one command to recover, one routine for conflicts.

Your win for today: rebase a feature branch to clean it up before a PR, knowing exactly how to undo it if anything looks wrong.

1 · The Golden Rule keeps you safe

The Golden Rule of Rebasing

Never rebase commits that have been pushed and shared — that others may have based work on.

Pro Git puts it bluntly: "Do not rebase commits that exist outside your repository and that people may have based work on… if you don't, people will hate you." [Pro Git §3.6]

This is the same private-vs-shared question from Lesson 2, now as a hard line. Rewriting a private commit is harmless — nobody else has it. Rewriting a shared one creates a divergent twin (C vs C' from Lesson 1), and now two people's histories disagree about reality. That's the mess.

The safe default

Rebase your own local branch to tidy it before pushing. Once it's pushed and someone else might be building on it, switch to merge. Best of both worlds.

2 · The reflog is your undo button

Here's the fear-killer. Rebase almost never truly destroys commits — it moves the branch pointer to the new copies and leaves the originals sitting unreferenced. Git remembers every place HEAD has been in the reflog for weeks. [git-reflog]

$ git reflog
a1b2c3d HEAD@{0}: rebase (finish): returning to refs/heads/feature
e4f5g6h HEAD@{1}: rebase (pick): D
1a2b3c4 HEAD@{2}: commit: D          <- here's my branch BEFORE the rebase
...

# "Take me back to exactly where I was before the rebase":
$ git reset --hard HEAD@{2}
Read this and the fear is gone

A rebase you don't like is not a catastrophe — it's one git reset --hard away from undone. The pre-rebase commit is still in the reflog. Knowing the exit exists is what lets you walk in calmly.

3 · Conflicts: pause, fix, continue (or abort)

Replaying a commit can hit a line the new base also changed. Rebase pauses and hands you the wheel. You are never stuck — there are exactly two ways out, and both are safe.

$ git rebase main
CONFLICT (content): Merge conflict in app.js
# rebase stops on the conflicting commit and waits.

# 1. Open the file, resolve the <<<</====/>>>> markers by hand.
$ git add app.js              # stage the resolution

$ git rebase --continue       # replay the next commit
#   ...repeat if more commits conflict.

--continue

You fixed the conflict and staged it. Resume replaying the remaining commits. The forward path.

--abort

Changed your mind, or the conflict is a mess. Instantly puts the branch back exactly as it was before you started. The escape hatch.

$ git rebase --abort          # nothing happened; branch restored to pre-rebase state

(There's also --skip to drop the current commit, but you rarely want it — reach for --abort and rethink instead.) [git-rebase]

The fearless-rebase checklist

This is the whole course, distilled into a routine. Run it the next time you clean up a branch before a PR (that's scenario ①).

  • Ask the deciding question. Are these commits still private to my local branch? If shared → stop, merge instead.
  • Know your shape. You're flattening a fork into a line (Lesson 1). Expect new hashes — that's normal, not loss.
  • Note your escape route. Before rebasing, glance at git log --oneline -1 (or trust the reflog) so you know where "before" is.
  • Rebase. git rebase main.
  • If it conflicts: resolve → git add → git rebase --continue. Too messy? git rebase --abort and nothing is lost.
  • Verify the graph. git log --graph --oneline --all — confirm a clean line on top of main.
  • Regret it? git reflog → git reset --hard HEAD@{n} back to before.
  • Push. Now it's shared — Golden Rule is in force; no more rebasing these commits.

The final self-check

From memory: after resolving a rebase conflict, what two commands move you forward?

git add <file> to stage your resolution, then git rebase --continue to replay the next commit. If it ever feels wrong, git rebase --abort puts everything back.
