Git merge vs. rebase · Lesson 3 of 3

Rebase without fear

You know the two shapes (Lesson 1) and how to choose (Lesson 2). The last thing standing between you and a confident rebase is fear: "I'll lose work." This lesson dismantles it with three things — one rule, one safety net, and one conflict workflow.

1 · The golden rule (your guardrail)

The Golden Rule of Rebasing Never rebase commits that exist outside your repository and that other people may have based work on.1 Rewrite only private, un-pushed history.

This is the same "are these commits shared?" question from Lesson 2, now stated as a safety rule. Obey it and rebase cannot hurt anyone else: you're only ever reshaping commits that are yours alone. The horror stories — clobbering a teammate's work — are all violations of this one rule.2

2 · The safety net: you can't actually lose committed work

Here's the fact that kills the fear. A rebase doesn't delete your old commits; it builds new ones and moves the branch pointer to them. The originals linger in the reflog — git's local record of every place HEAD has been — for about 90 days.3

git reflog
# e3a1b2c HEAD@{0}: rebase finished: returning to refs/heads/feature
# 9f8e7d6 HEAD@{1}: rebase: My second commit
# 1a2b3c4 HEAD@{2}: checkout: moving to main
# 7c6b5a4 HEAD@{3}: commit: My second commit   ← the ORIGINAL, pre-rebase tip
#
git reset --hard HEAD@{3}    # branch pointer jumps back; work restored
Reframe A rebase you regret is undoable. Find the pre-rebase line in git reflog, git reset --hard to it, and you're exactly where you started. "Rebase loses work" is false for committed work.

3 · The conflict workflow (--continue / --abort)

Because rebase replays commits one at a time, a conflict can surface at each one. Git pauses and hands you the wheel. You have three buttons:4

Command Meaning
git rebase --continue "Fixed it — replay the next commit." (After git add.)
git rebase --abort "Forget the whole thing — put my branch back exactly as it was."
git rebase --skip "Drop this one commit and keep going." (Use rarely, deliberately.)
git rebase main
# CONFLICT (content): Merge conflict in app.js
#
# 1. open app.js, resolve the  <<<<<<<  =======  >>>>>>>  markers
# 2. git add app.js
# 3. git rebase --continue          # the next commit replays
#
# …repeat for each conflicting commit, OR at any moment:
git rebase --abort                  # bail out, zero damage
The escape hatch removes the risk --abort always exists. If a rebase gets confusing, you are never trapped — one command restores the original branch. Knowing the exit is what lets you walk in calmly.

The applied checklist

This is your whole course on one card. Run it before any rebase:

  • Shared? Has anyone pulled these commits? If yes → stop, use merge (Lessons 2 & the golden rule).
  • Private & want it clean? → git rebase main. Expect new hashes (Lesson 1).
  • Conflict appears? Resolve → git add → git rebase --continue. Confused at any point → git rebase --abort.
  • Regret the result? git reflog → git reset --hard HEAD@{n}. Nothing committed is truly lost.
  • Never force-push a rebased shared branch.
