Rebase Without Fear

Rebase fear comes from two sources: "I'll lose work" and "I'll break a teammate's history." Both are real risks — and both have concrete safeguards. This lesson gives you those safeguards and the workflow to apply them every time.

The golden rule

Never rebase commits that exist outside your local repository.

More precisely: never rebase commits that someone else might have based work on. (Pro Git §3.6: The Perils of Rebasing)

Here is why. When you rebase, git replaces your commits with new ones that have new hashes (as you saw in Lesson 1). If a teammate has already pulled the original commits and based new work on them, their history and yours now contain two different commits that both claim to descend from the same ancestor. When they pull again, git sees two divergent branches where there should be one. Untangling that is genuinely painful.

The safeguard is simple: before rebasing, ask whether the branch is shared. If yes — use merge instead. If it is truly yours alone, rebase freely.

Your safety net: git reflog

The reflog is a local journal of every position HEAD has ever pointed to, including positions that no longer exist on any branch. (git-reflog docs) When you rebase, your original commits are not deleted — the branch pointer simply moves to new commits. The reflog still knows where you were. 

$ git reflog

3f1a2c0 HEAD@{0}: rebase (finish): returning to refs/heads/feature
8b3e4d1 HEAD@{1}: rebase (pick): E — add login form
2c7f9a3 HEAD@{2}: rebase (pick): D — scaffold auth routes
e4a1b0c HEAD@{3}: checkout: moving from feature to feature
a92d1f5 HEAD@{4}: commit: E — add login form    ← original E
71bc30e HEAD@{5}: commit: D — scaffold auth routes  ← original D

To undo the entire rebase and return to exactly where you were:

$ git reflog                      # find the entry just before the rebase started
$ git reset --hard HEAD@{4}       # or paste the hash directly: git reset --hard a92d1f5

This is not a workaround — it is the designed recovery path. Knowing it exists turns "rebase is dangerous" into "rebase is reversible."

The reflog is your parachute. It records every state HEAD has been in, for as long as the entry is retained (90 days by default). A rebase gone wrong is almost always undoable with the reflog.

Resolving conflicts mid-rebase

When git replays your commits onto the new base, it may pause at a commit where your changes conflict with what's already there. This is identical to a merge conflict — same markers, same tools — with one difference in how you signal completion.

What it looks like

$ git rebase main

Auto-merging src/auth/routes.js
CONFLICT (content): Merge conflict in src/auth/routes.js
error: could not apply a92d1f5... D — scaffold auth routes
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".

Resolution steps

  1. Open the conflicted file. Look for conflict markers: 
    <<<<<<< HEAD
// code from main (the new base)
=======
// code from your commit D
>>>>>>> a92d1f5 D — scaffold auth routes
  2. Edit the file to what it should be. Delete all three marker lines (<<<, ===, >>>) and keep only the correct code.
  3. Stage the resolved file — do not create a commit: 
    $ git add src/auth/routes.js
  4. Continue the rebase: 
    $ git rebase --continue
    Git moves to the next commit. If that commit also conflicts, the cycle repeats. (git-rebase docs: --continue)

Abort cleanly if you get stuck

If at any point you want to abandon the rebase and return to your original branch state: 

$ git rebase --abort

--abort is a clean, complete undo — your branch returns to exactly what it was before you started the rebase. Use it freely if a conflict overwhelms you; you can always start again when you are ready.

Do not run git commit during a rebase. When resolving a conflict, stage the file with git add and then run git rebase --continue. Running git commit creates an unexpected extra commit and disrupts the rebase sequence.

Pushing a rebased branch

After rebasing, your local branch has diverged from the remote — your new commits don't share ancestry with the old ones on the server. A plain git push will be rejected. You need to force-push: 

$ git push --force-with-lease

--force-with-lease is safer than --force. It checks that the remote tip matches what you last fetched. If someone else has pushed to your branch since your last fetch, --force-with-lease aborts rather than silently overwriting their work. (git-push docs: --force-with-lease) Use it as your default for all post-rebase pushes.

Safe rebase workflow

Apply this checklist before every rebase. The first gate is the only one that can't be bypassed.

  • Is this branch pushed to a shared remote where a teammate could have pulled from it?
    If yes: use git merge main instead. Stop here.
    If no: continue.
  • Run the rebase:
    git rebase main
  • Conflict? For each conflicted file:
    edit the file → git add <file> → git rebase --continue
  • Stuck or something looks wrong?
    git rebase --abort — restores your branch completely
  • Rebase succeeded but you want to undo it?
    git reflog → find the commit just before the rebase → git reset --hard <hash>
  • Push the rebased branch:
    git push --force-with-lease

Putting it all together

These questions interleave material from all three lessons. The goal is to build connected recall, not isolated facts.

You just ran git rebase main and the result looks wrong. The rebase completed — there is no rebase in progress. What is the first command you run to undo it?

git reflog is the right first move. Find the entry labeled something like commit: E — your last real commit from before the rebase started, then git reset --hard <hash>. (git rebase --abort only works during a rebase; once it completes, there is nothing to abort.)

Mid-rebase, git pauses with a conflict in src/auth.js. After you edit the file to resolve it, what is the correct sequence of commands?

Stage with git add, then git rebase --continue. Do not run git commit — that would insert an unexpected extra commit into the rebase sequence. Rebase creates the commit for you when you call --continue.

You pushed your feature branch yesterday. A colleague mentioned they cloned the repo and checked out your branch to try something. Today you want to rebase it. What do you do?

The moment someone else checks out your branch, it is effectively shared. Rebasing and force-pushing rewrites commits they have based work on — leaving their local branch diverged with no clean path forward. Merge is the safe choice, even if the resulting graph is less tidy.

After a successful rebase, you try git push. Git rejects it. What is the safest command to push the rebased branch?

--force-with-lease is the right tool. It behaves like --force but adds one safety check: if someone else has pushed to the remote branch since your last fetch, it aborts. This protects you from silently overwriting a teammate's accidental push to your branch. Make it your habit whenever you force-push after a rebase.

What does git reflog show that git log does not?

The reflog records HEAD movements, not just reachable commits. Commits that have been abandoned by a rebase — no longer pointed to by any branch or tag — disappear from git log but remain in git reflog. That is what makes it the recovery tool for a rebase gone wrong.
Primary source for this lesson git-rebase — Official Reference Manual (git-scm.com) — the authoritative flag-by-flag reference for every option covered here: --continue, --abort, --force-with-lease. Bookmark it for the edge cases you will inevitably hit in production.
