Git merge vs. rebase · Lesson 1 of 3

The two shapes of history

You merge everything because rebase sounds dangerous. Before any decision-making, you need one true picture in your head: what each command actually does to the commit graph. Get this, and the fear mostly evaporates — it was fear of the unknown.

This lesson has exactly one win: you'll be able to draw the graph that merge produces and the graph that rebase produces, and say why rebase changes commit hashes. No choosing between them yet — that's Lesson 2.

The starting point

A commit is a snapshot plus a pointer to its parent, named by a hash.1 A branch is just a movable pointer to one commit. Say you branched feature off main, you each made commits, and now the histories have diverged:

A───B───C ← main \ D───E ← feature (your work)

Both merge and rebase answer the same question — "how do I combine these?" — but they produce two different shapes.

What merge does

git merge main (run from feature) creates a brand-new merge commit — call it M — that has two parents, one from each line. Nothing that already existed is changed; the real, branching history is preserved exactly as it happened.2

A───B───C───────M ← feature \ / D───E───╯

M is the visible signature of a merge: the one commit in the graph with two parents. Commits D and E keep their original hashes. History is a record of what happened.

What rebase does

git rebase main (run from feature) does something different: it takes your commits D and E, sets them aside, moves to the tip of main, and replays them one at a time on top of C.3 The result is a straight line — no merge commit:

A───B───C───D′───E′ ← feature
Why the primes (D′, E′)? D′ and E′ are new commits with new hashes. A commit's hash is computed from its content and its parent. Replay gives D a new parent (C instead of B), so it must become a new commit. Same changes, new identity. The originals aren't edited — they're abandoned (and, crucially, still recoverable — Lesson 3).

See them side by side

merge

A─B─C───M \ / D─E╯
  • Keeps both lines
  • Adds a merge commit
  • Hashes unchanged
  • History = record

rebase

A─B─C─D′─E′
  • One straight line
  • No merge commit
  • New hashes (D′, E′)
  • History = story

Both end with the same files — the same final code.2 They differ only in the shape of the history they leave behind. That difference is the whole game.

Recall — do it from memory

Don't just nod along; retrieval is what makes this stick.2 Try each before revealing.

Draw the graph merge produces from the diverged start. What's the tell-tale feature?
A─B─C───M ← feature \ / D─E╯
The tell-tale: a merge commit M with two parents. Original commits keep their hashes.
Draw the graph rebase produces. Why can't the commits keep their old hashes?
A─B─C─D′─E′ ← feature
A straight line, no merge commit. The hashes change because a hash is derived from the commit's content plus its parent; replay gives each commit a new parent, so it's a new commit.

Your tangible win

You can now Draw both shapes from a diverged start, and explain that rebase produces new hashes because each replayed commit gets a new parent. That's the mental model the rest of the course builds on.
