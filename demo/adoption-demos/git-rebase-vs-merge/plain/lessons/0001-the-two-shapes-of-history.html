The Two Shapes of History

Before choosing between merge and rebase, you need to see exactly what each one does to the commit graph. That is the only goal of this lesson.

Git history is a graph

Every git repository is a directed acyclic graph (DAG) of commits. Each commit stores a snapshot of your files and a pointer to its parent commit. When you create a branch, you fork that graph: 

A ── B ── C          ← main
          \
           D ── E    ← feature

You are on feature. Since you branched, main has moved forward to C. You need to integrate these two lines of work. You have exactly two choices.

Shape 1 — Merge

From main, run git merge feature. Git creates a merge commit — a new commit M with two parents: C and E. 

A ── B ── C ── M     ← main (HEAD)
          \   /
           D ─ E     ← feature

Three things to observe:

  • M has two parents. A merge commit is the only kind of commit with more than one parent. That is its defining characteristic. (Pro Git §3.2)
  • D and E are unchanged. Their content, author timestamps, and SHA-1 hashes are identical to before the merge. Merge never rewrites existing commits.
  • The graph has a diamond shape. git log --graph will show two lines converging at M — the branch and rejoin are part of the permanent record.
Merge is purely additive. It only creates; it never modifies or destroys existing commits.

Shape 2 — Rebase

From feature, run git rebase main. Git takes each of your commits and replays them one by one on top of C, producing brand-new commit objects. 

Before rebase:                     After rebase:

A ── B ── C          (main)        A ── B ── C          (main)
          \                                  \
           D ── E    (feature)                D' ── E'   (feature)

D′ and E′ are entirely new commits. They apply the same changes as D and E, but they have a different parent (C instead of B), different timestamps, and therefore completely different SHA-1 hashes. (Pro Git §3.6)

Three things to observe:

  • New hashes. D′ ≠ D. Same diff, different commit object. If you printed both hashes, they would not match a single character.
  • No merge commit. The history is linear — every commit in the graph has exactly one parent. Running git log --graph shows a straight line.
  • The original D and E still exist — in your reflog — until git garbage-collects them. Nothing is truly deleted immediately. (The reflog is your safety net; you will use it in Lesson 3.)
Key implication Because rebase produces new commits, if anyone else has based work on your original D or E, their history and yours will diverge. That is the danger rebase is famous for — and it is avoidable with one rule, covered in Lesson 3.

At a glance

Merge Rebase
Creates a merge commit? Yes — with two parents No
Existing commits changed? No — hashes are preserved Yes — new hashes for every replayed commit
History shape Branching graph (diamond) Linear
Rewrites history? No Yes

Check your understanding

Each question has exactly one correct answer. Click to see feedback immediately.

After git rebase main, the commits on your feature branch have…

Correct. Rebase replays your diffs as fresh commit objects. A commit's hash is derived from its content, its author, its timestamp, and its parent hash. Because the parent changes, every downstream hash changes — even if the diff is byte-for-byte identical.

A merge commit is special because it…

Correct. The defining property of a merge commit is its two parent pointers — one to the tip of each branch that was merged. That is what preserves the branching graph in the permanent history.

You run git log --graph after integrating two branches. The output is a perfectly straight line — no diamonds, no branches. Which operation was used?

Correct. After rebasing, the feature branch tip is a direct descendant of main, so main can fast-forward to it without creating a merge commit — leaving a straight line. (A plain git merge with fast-forward enabled would also be linear, but only if the feature branch was a strict descendant of main already — i.e. main hadn't moved, which contradicts the scenario.)

After a rebase, what happens to the original commits (D and E in the diagram above)?

Correct. Git never immediately destroys unreachable commits. They remain accessible via git reflog for as long as the reflog retains them (typically 90 days by default). This is why a rebase gone wrong is recoverable — a point you will put into practice in Lesson 3.
